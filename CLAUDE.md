# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Whispers In The Void — a procedurally generated 3D space trading/combat/exploration game built with Electron + Three.js (WebGL), no UI framework (plain DOM overlays for HUD/menus/trading). Real-time arcade flight/combat, one seeded galaxy of ~1500 planets and 100 ship classes, permadeath.

## Commands

```
npm run dev       # electron-vite dev — launches the app with HMR
npm run build     # electron-vite build
npm run package   # build + electron-builder --dir (unpacked)
npm run make      # build + electron-builder (installer for the current platform)
```

There is no `npm test` script. Tests use Node's built-in test runner directly, colocated as `*.test.js` next to the module they cover:

```
node --test src/renderer/game/*.test.js src/renderer/procgen/*.test.js src/renderer/data/*.test.js   # everything
node --test src/renderer/game/combat.test.js                              # single file
```

Editing `src/renderer/main.js` while `npm run dev` is running triggers HMR, which resets the running game back to the main menu — expected, not a bug.

To drive the app programmatically (e.g. for scripted verification), launch with `DEV_REMOTE_DEBUG=<port> npm run dev`; this appends `--remote-debugging-port` in `src/main/index.js` so it can be attached to over the Chrome DevTools Protocol.

## Architecture

### Process split

- `src/main` — `index.js` creates the `BrowserWindow`; `ipc.js` registers `save-game`/`load-game`/`delete-save`/`has-save`/`quit-app` handlers backed by `save.js` (plain `fs` read/write of JSON to `app.getPath('userData')/saves/save1.json`). `window-all-closed` always calls `app.quit()`, on every platform including macOS — Electron's own boilerplate skips this on darwin (the multi-document-editor convention of staying alive in the dock after the last window closes), but that doesn't fit a single-window arcade game with nothing useful to do in the background; without the override, closing the window (red button or Cmd+W) left the whole process tree running indefinitely on macOS. Verified directly: launched a dev instance, called `window.close()` on its page via CDP (the same path the red button takes), and confirmed every Electron process (main, GPU, renderer, utility helpers) exited.
- `src/preload` — `contextBridge` exposes only `saveGame`/`loadGame`/`deleteSave`/`hasSave`/`quitApp` as `window.electronAPI` (`contextIsolation: true`, `nodeIntegration: false`).
- `src/renderer` — the entire game: Three.js scene plus DOM overlays. Everything below is under here.

### Renderer layout

- `procgen/` — pure, seeded generation: `prng.js` (`mulberry32` + `pick`/`range`/`intRange` helpers), `galaxy.js` (systems and their bodies), `names.js`, `hull.js` (parametric ship hull geometry — cross-section "stations" lofted into a `BufferGeometry`, plus optional wings), `shipRoster.js` (procedurally generates the bulk of the ship classes), `starType.js` (`STAR_TYPES` + `starTypeForSystem` — shared by `starMesh.js` and starting-system pick so both agree which systems are binary).
- `data/` — mostly static tables: `shipClasses.js` (7 hand-crafted classes in `HAND_CRAFTED_SHIP_CLASSES` + 93 more from `generateShipClassRoster`, seeded, for 100 total), `goods.js` (includes `MINED_ORE_GOOD_IDS`, kept out of the regular cargo trade table — see mining below), `weapons.js` (buyable hardpoint weapons — see Weapons & the armory below), `economyTags.js`, `missionTemplates.js` (one generator function per mission type, including `probe`).
- `game/` — pure gameplay logic operating on plain data via a `gameState` object (no direct Three.js scene/mesh references, only `THREE.Vector3`/`Quaternion` math): `state.js`, `flight.js`, `combat.js`, `economy.js` (trading, repair, per-station storage transfers, ship parts), `missions.js` (accept/turn-in/progress plus `missionNavTarget`/`missionMarkedSystemIds`/`setWaypointForMission` for the missions UI and map markers), `hyperspace.js`, `supercruise.js`, `spawner.js`, `probe.js` (random survey-data find on probing a body), `mining.js` (mining an asteroid field for ore — triggered by `combat.js` when a player projectile hits an `asteroidField` body, not a dedicated key), `collision.js` (`resolveBodyCollisions`: sphere-sphere push-out + inward-velocity cancel against system bodies — no damage, just deflection), `wrecks.js` (lootable wrecks left by destroyed ships), `save.js` (serialize/deserialize `gameState`, calls `window.electronAPI`).
- `render/` — Three.js mesh builders and scene/camera helpers: `scene.js`, `starfield.js`, `shipMesh.js`, `stationMesh.js`, `stationInterior.js` (generic docking-bay interior shown behind the docking UI), `planetMesh.js`, `starMesh.js`, `asteroidFieldMesh.js`, `wreckMesh.js`, `projectileMesh.js`, `thrusterParticles.js`, `damageEffects.js`, `sceneSync.js`, `lensFlare.js`, `nebula.js`, `textures.js`.
- `ui/` — DOM overlays: `menu.js`, `hud.js`, `dockingUI.js`, `navMap.js`, `pauseMenu.js`, `inventoryUI.js`, `missionsUI.js` (active-mission tracker, **J**), `deathScreen.js`.
- `main.js` — the orchestrator. Owns the `animate()` frame loop, wires `game/` logic to `render/` meshes and `ui/` overlays, and owns session lifecycle (`startSession`/`clearSession`/`returnToMenu`), docking eligibility, hyperspace/supercruise effect state machines, mission waypoint resolution (`getActiveWaypoint`), and radar contact computation.

### `gameState`

A single mutable object created by `game/state.js` and passed as the first argument through nearly every `game/` function. Holds `player` (ship, credits, reputation, `currentSystemId`, `startingSystemId`, `waypointBodyId`, optional free-space `waypointPosition` for bounty hunt markers), `galaxy` (systems/bodies from procgen), `missions` (`available`/`active`), `npcs`, `projectiles`, `wrecks` (lootable wrecks — see Wrecks & salvage below), `stationStorage` (per-body storage — see Per-station storage below), `economyOverrides`, `visitedBodyIds`, `probedBodyIds` (distinct from `visitedBodyIds` — set only when a body is actually probed via `game/probe.js`, not merely flown near or docked at; this is what lets `probe`-type missions require an actual probe action), `inCombat`, `simTime`, `flags` (`alive`, `startingSystemPeaceBroken`). The player's `ship` has two separate hold pools — `cargo` (regular trade goods, capacity `cargoCapacity`) and `miningHold` (mined ore only, capacity `miningCapacity`) — kept apart so mining never interacts with regular cargo trading — plus a `shipParts` count (a rare consumable, not a cargo slot — see Ship parts below). `game/save.js`'s `serializeGameState` defines exactly what gets persisted — `npcs`, `projectiles`, and `wrecks` are deliberately excluded (see Save/load below); `stationStorage` is persisted.

### Timing

All cooldowns, shield regen, and combat/AI timers are driven by `gameState.simTime` (seconds accumulated from per-frame `dt` in `main.js`'s `animate()`), never wall-clock time. This keeps combat logic deterministic and testable independent of frame rate — see `combat.js`.

### Ship/camera coordinate convention

This is non-obvious and spans several files — read this before touching flight, combat AI, or the camera:

- Ships treat local **+Z as forward**, +X as right, +Y as up (right-handed: right×up=forward). Used throughout `flight.js`, `combat.js`, `hyperspace.js`, `supercruise.js`, and hardpoint position data.
- `THREE.Matrix4.lookAt(target, source, up)` is deliberately called with the eye/target arguments swapped in `combat.js` (NPC facing) and `supercruise.js` (autopilot facing), because Three.js's `lookAt` follows the camera convention (local +Z points *away* from the target) while ships use local +Z as forward.
- The chase camera (`render/sceneSync.js`'s `syncChaseCamera`) sits behind and above the ship via `camera.lookAt()` with a world-up vector. Because camera space is -Z-forward (the opposite of the ship's +Z-forward), the camera's actual world-space "right" ends up equal to the ship's **local -X**. This is why `main.js`'s `computeRadarContacts()` negates `rel.x` — the comment there has the full derivation.

### Procedural generation

- A single `mulberry32(seed)` PRNG drives all galaxy generation (system layout, bodies, tags, names, missions) so a seed reproduces the same galaxy. Save files store the fully-generated galaxy array directly rather than replaying the RNG.
- Visuals/physics that don't need their own persisted field (star size/hue, asteroid rock scatter within a field) are derived from a deterministic `hashString(body.id)` hash instead. Planet/moon/asteroidField size is the exception: it's rolled once at generation time and stored as `body.radius`, since it's both a render size and a collision size (see `game/collision.js`) and needs to agree between the two.
- Body kinds: `planet`, `moon` (orbits close to one specific parent planet, referenced by `parentId`; ~23% of planets have one), `station`, `settlement`, `asteroidField` (scattered across a small fraction of systems). Only `station`/`settlement` are always dockable; a `planet`/`moon` is dockable only if it has a base, which reuses the existing `hasMissions` flag as that signal (see `isDockable` in `main.js`). `asteroidField` is never dockable, but is mineable (see below); `planet`/`moon`/`asteroidField` are all probeable regardless of whether they have a base.
- Galaxy scale is controlled by `generateGalaxy`'s default `opts` in `galaxy.js`: 450 systems, 1500 planets, 180 stations, 120 settlements, 40 asteroid fields. Ship class count is controlled the same way in `data/shipClasses.js` (7 hand-crafted + a generated count passed to `generateShipClassRoster`, currently 93, for 100 total).
- Every ship class has a `stats.miningCapacity` (mining hold size), sized off price and role via `computeMiningCapacity` in `procgen/shipRoster.js` (floor `MINING_HOLD_MIN`, 30) — the starter ship (`bravia_mk2`) is hard-set below that formula's floor (`STARTER_MINING_CAPACITY`, 15, in `data/shipClasses.js`) so it's always the smallest hold in the game. Both were tripled together (from 10/5) per a later balance pass, keeping the same 2:1 floor-to-starter ratio so every other ship's hold kept scaling off the new base rather than drifting back toward the old floor.
- Mining a rock (`game/mining.js`'s `mineRock`) yields one of four ore tiers (`MINED_ORE_GOOD_IDS` in `data/goods.js`) chosen by the *system's* distance from the galaxy's core (`coreFraction(system)` in `procgen/galaxy.js`, 0 at the core / 1 at the rim) — systems near the rim yield rarer, more valuable ore. Triggered by firing at an individual rock, hit-tested the same way Tab-targeting resolves one (`render/asteroidFieldMesh.js`'s `getAsteroidRocks`, see `combat.js`'s `updateProjectiles`), not the whole field's bounding sphere.
- Each rock holds a finite 10-200 units of ore, rolled once off a deterministic per-rock seed (`hashString('fieldId:index')`, the same convention `getAsteroidRocks` itself uses) and stored as mutable state in `gameState.asteroids` (keyed `"fieldId:index"`, lazily created by `game/mining.js`'s `rockState`) since depletion has to actually persist as it's mined down — unlike the rest of a field's per-rock data (position/size/rotation), which stays a pure function of that same hash. Emptying a rock "explodes" it (a bigger impact flash plus `playExplosion`, see `main.js`'s `onProjectileHit`) and hides its mesh (`animate()`'s per-frame `isRockAlive` check over each asteroid field's `bodyMeshes` children) until a second per-rock-seeded roll — 12-24hrs of accumulated `simTime`, the same "active-play-time" convention as `game/wrecks.js`'s `WRECK_DESPAWN_S` — elapses, at which point it quietly comes back at full ore. Deliberately ephemeral like wrecks/npcs (`gameState.asteroids` is reset on load, never serialized — see `game/save.js`), both to match that convention and because `gameState.simTime` itself resets to 0 on load, which would otherwise make a persisted respawn timer meaningless across a save/reload. A rock's target/HUD label (`main.js`'s `resolveTarget`) is named after its ore tier via `mining.js`'s `rockDisplayName` (e.g. "Raw Ore Deposit") rather than the field's own generated Belt name, and disappears from Tab-targeting entirely while destroyed.
- `procgen/galaxy.js`'s `radiusFor`/`localPosition` apply `PLANET_SIZE_SCALE` (37.5x), `MOON_SIZE_SCALE` (24.75x), and `SYSTEM_SIZE_SCALE` (9x, applied to `SYSTEM_LOCAL_MIN/MAX_RADIUS`) on top of the base rolled ranges; stations are scaled 11.25x (`STATION_SCALE` in `main.js`, plus a small per-body +/-15% render-only variance from the existing `hashStringForOrbit` hash) and suns 37.5x (`STAR_SIZE_SCALE` in `render/starMesh.js`). These went through three rounds of scale-up (2.5x/1.65x/5x/1.5x/2.5x → 12.5x/8.25x/7x/4.5x/12.5x → the current values) since each pass still read as too small once compared side-by-side — `SYSTEM_SIZE_SCALE` climbed alongside each round (5x→7x→9x) purely to give the increasingly bigger bodies room to spread out without visually crowding. A per-system `sizeScale` (`SYSTEM_SCALE_VARIANCE`, 0.85-1.15) is rolled once per system in `generateGalaxy` and threaded into every body placed in it (including stations/settlements/asteroid fields added after the fact, via `system.sizeScale`), so systems vary in overall size rather than all reading as uniform. `game/collision.js`'s fixed `STATION_COLLISION_RADIUS`/`SETTLEMENT_COLLISION_RADIUS` are scaled 11.25x to match (the per-body render variance isn't mirrored there — `main.js`'s `dockRangeFor` margin absorbs that small a mismatch).
- Both `render/planetMesh.js` and `render/starMesh.js` build their bodies' surfaces on a radius-scaled `IcosahedronGeometry` via `detailForRadius` (planets/moons up to 6, star cores up to 6). Planet materials use **smooth** shading (`flatShading: false`) and no `EdgesGeometry` overlay — flat shading + edge lines re-fragmented large worlds even after detail steps. Surface interest comes from vertex colors + PBR maps. Cheap: mesh build runs once when entering a system, never per frame.
- `coreFraction(system)` (`procgen/galaxy.js`) is the shared 0-1 "how far toward the rim" helper — besides ore tier, `game/spawner.js`'s `spawnEncounterNear` takes it as `alienBias` to zero out alien encounters at the core and ramp them up toward the rim (pirate odds stay fixed; trader share fills the rest), and `main.js`'s ambient-spawn loop uses it to raise the NPC cap and shorten the respawn interval near the core (busier) and do the opposite toward the rim (quieter) — implementing the "more populated near the core, more alien activity toward the rim" gradient. `game/state.js`'s `pickStartingSystem` uses the same helper to start new games in one of the 10% of systems closest to the core that are **not** binary stars (`starTypeForSystem` from `procgen/starType.js`, same type table `starMesh.js` uses).
- Hyperspace/new-game arrival (`SYSTEM_ARRIVAL_POSITION` in `galaxy.js`, used by both `game/state.js` and `game/hyperspace.js`) sits near the (now much larger) system's edge on -Z, scaled off `SYSTEM_LOCAL_MAX_RADIUS`. `hyperspaceJump` additionally resets the ship's quaternion to identity on arrival — since local +Z is forward and the arrival point is on -Z, this always faces the star at the system origin regardless of which way the ship was facing before the jump (new-game start already defaults to identity, so it was already correct there).
- Hyperspace only reaches "neighboring" systems, not anywhere on the map — crossing the galaxy means hopping system to system. `galaxy.js`'s `computeNeighborLanes` (called once at the end of `generateGalaxy`) gives every system a `neighborIds` list: its `JUMP_NEIGHBOR_COUNT` (5) nearest systems by galaxy-map distance, then adds the reverse edge for each (if A lists B, B also lists A) so a reachable system can always be jumped back from — a plain distance cutoff was rejected because it could leave an outlier system with zero systems in range, permanently stranding the player. `canJumpTo(fromSystem, toSystemId)` is the single check shared by `game/hyperspace.js` (enforcement), `main.js`'s `handleJump` (pre-validation, so the jump animation doesn't play just to fail partway through), and `ui/navMap.js` (the galaxy map draws lane lines from the current system to each neighbor and greys out/disables the jump button for anything else).

### Body/star visuals

- `render/planetMesh.js`, `render/starMesh.js`, and `render/stationMesh.js` all use a cheap seeded "value noise" — `Math.sin(nx*3.7+offsets[0]) * Math.cos(ny*4.1+offsets[1]) * Math.sin(nz*3.3+offsets[2])` evaluated on each vertex's normalized position — to get smooth, continuous surface blotches (craters, oceans, lava cracks, flare patches) from vertex colors alone, without a texture/noise library. Per-vertex independent randomness was rejected early on since it reads as uncorrelated static rather than a texture.
- `buildPlanetMesh` picks one of five archetype builders (`buildRocky`, `buildGasGiant`, `buildIce`, `buildLush`, `buildVolcanic`) seeded off `hashString(body.id)`, so the same body always renders the same way; moons always use a separate `buildMoon` (small cratered rock only — bands/oceans/lava don't make sense at that scale). Archetypes combine the noise field with `jitterGeometry` (radial vertex displacement + `computeVertexNormals()`) for bumpiness, except the gas giant, which stays smooth and uses latitude-based sine bands instead of the noise field.
- `render/textures.js`'s `getSurfaceTextures(archetype)` layers real photo detail on top of the vertex-color noise above rather than replacing it: four CC0 (public domain, no attribution required) PBR photo sets from [ambientCG](https://ambientcg.com/) — `rock` (Rock064), `ice` (Snow006), `lush` (Moss002), `lava` (Lava005), each a `{map, normalMap, roughnessMap}` triple under `src/renderer/public/textures/` — are tiled via `RepeatWrapping` (`repeat.set(4,2)`) across each body's low-poly icosahedron rather than wrapped as one fixed equirectangular image, since these are seamless square swatches, not whole-globe maps; `IcosahedronGeometry`'s own built-in spherical UV unwrap is enough to tile them reasonably. `gasGiant` has no entry (cloud bands don't suit a tiled rock/ice/lava photo), so `getSurfaceTextures('gasGiant')` returns `undefined`, which every caller spreads harmlessly (`{...undefined}` is a no-op) into its material options. Every body of a given archetype shares the *same* three texture objects rather than cloning one per body — GPU memory stays bounded regardless of galaxy size (up to ~1500 planets, though only ever a handful are live at once since `main.js` tears down the previous system's meshes on jump/dock) — so per-body variety still comes entirely from the existing vertex-color tint, not from the texture itself. Loading is lazy (deferred to first actual use inside a mesh-builder function, not at module import time) specifically because `game/combat.js`/`game/missions.js`'s `node --test` suites transitively import this module (via `asteroidFieldMesh.js`'s `getAsteroidRocks`) but run under plain Node with no `document` — `THREE.TextureLoader` would throw immediately if a texture load were triggered eagerly at import time. `buildPlanetMesh`'s materials switched from `MeshLambertMaterial` to `MeshStandardMaterial` to support `normalMap`/`roughnessMap` (real lit relief, not just flat vertex-color blotches); moons and asteroid rocks (`asteroidFieldMesh.js`) both reuse the `rocky` set (bare rock doesn't need its own archetype variety). The sun's core (`buildSingleStar`) only takes the `volcanic` set's `map` — no `normalMap`, since `MeshBasicMaterial` (its self-lit, ignores-scene-lighting material — see below) never samples one, and switching to a lit material risked the "roiling plasma, not a flat gem" look plus the lens flare's self-occlusion fix (see the next bullet) for a detail that wouldn't show through the additive corona shells most of the time anyway.
- ~3% of planets (never moons) also get a tilted ring (`buildRing`, a banded-color `RingGeometry` disc) — like the archetype choice, this is decided by drawing one more value off the same per-body `rng` after the base mesh is built, not a persisted field, since it's purely cosmetic and the rng sequence is already deterministic per `body.id`.
- `render/lensFlare.js`'s `buildLensFlare(color)` wraps three.js's own `Lensflare`/`LensflareElement` addon (bundled with the installed `three` package, not a new dependency) rather than hand-rolled screen-space math — its built-in occlusion sampling already does exactly "fade out when off-screen or blocked by nearer geometry," which is the entire "looking directly at the sun" behavior, for free. Its element textures are baked once from Canvas 2D (the same technique `nebula.js`/`thrusterParticles.js` already use) and mimic a real lens: the source glow plus a blue-white anamorphic streak sit at the light itself, then a chain of soft-edged *hexagonal* aperture-blade ghosts (drawn as a 6-sided path with a bright-rim radial gradient — round ghosts only happen wide-open on real lenses) marches along the light-to-screen-center axis with the chromatic teal/green/purple/orange tints coating reflections pick up, growing bigger and fainter with distance, plus one big faint halo ring past center. `buildSingleStar` adds one as a child of the star's own `THREE.Group` at local origin (same as the corona shells), so it inherits that star's world position automatically — including a binary's orbiting component — with no per-frame code of its own. This runs for every star everywhere `buildStarMesh` is used, so in-system suns and the main-menu binary star both get it. Needed a `resolve.dedupe: ['three']` added to both `electron.vite.config.mjs` (the real app) and a small `src/renderer/vite.config.js` (only read by the standalone `vite` dev server used for browser-preview testing, per `.claude/launch.json`) — `three/examples/jsm/...` otherwise resolves `three` through a separate module graph than the rest of the app under Vite, triggering "Multiple instances of Three.js" and silently breaking the flare (its internals only recognize objects built from the *same* THREE classes as the renderer).
- `buildStarMesh`/`updateStarMesh` in `starMesh.js`: each star's core is a `MeshBasicMaterial` (self-lit, ignores scene lighting) over a noise-jittered, vertex-colored icosahedron so it reads as roiling plasma rather than a flat gem, plus two translucent additive corona shells (subdivision detail 3 — at 1 their handful of huge facets read as a crude octagonal frame once the halo brightened the star's surroundings). Each star also gets a big camera-facing halo `THREE.Sprite` (a lazy shared canvas radial-gradient texture, ~5x core radius, breathing slightly out of phase with the coronas) — this, more than the corona shells, is what makes a star read as blindingly bright — and its core's lava map is `clone()`d per star (not the shared instance from `render/textures.js`) so `updateStarMesh` can drift that star's `map.offset` every frame for a boiling-surface look without scrolling every volcanic planet too. `buildSingleStar` multiplies every `STAR_TYPE_PARAMS` radius by `STAR_SIZE_SCALE` (see Procedural generation above for its current value and history) in one place rather than editing each type's range — corona scales are already relative to radius so they grow proportionally for free. `buildTurbulentSurface`'s bump amplitude and hot-color contrast (`Math.pow(n, 0.7)` instead of a linear lerp) were both raised, and the corona pulse amplitudes in `updateStarMesh` bumped up, for a more fiery/dramatic look across every star type, not just the binary ring below. A system's star is one of `STAR_TYPES` (`mainSequence`, `redDwarf`, `whiteDwarf`, `giant`, or `binary`), picked by hash off `system.id`; each non-binary type is just a different `STAR_TYPE_PARAMS` entry (radius range, hue, corona opacity/scale) feeding the same `buildSingleStar` builder. `binary` calls `buildSingleStar` twice (from `BINARY_COMPONENT_TYPES`, which excludes `giant`/`binary` to avoid runaway recursion/scale); the bigger of the two stays stationary at the system origin (the "primary") and the smaller one orbits it — real binaries orbit a shared barycenter, but a single stationary anchor is simpler and keeps every other body's already-origin-relative position untouched. Separation is sized off *corona* reach (`(r0+r1)*2.2`, not just core radius), since corona shells extend up to ~1.9x a star's radius and a tighter separation just blends the two into one blob. `updateStarMesh(mesh, elapsed, dt)` — called each frame from `main.js` with `gameState.simTime` — iterates `mesh.userData.stars`, spinning each star's own core and pulsing its own corona; only the smaller star carries an `orbit` (the bigger one has none, so it just sits still). A binary pair also gets a fiery `energyRing` (`buildEnergyRing`): two concentric additive tori in the classic fire structure — a thin near-white inner core torus inside the fat deep-red-to-orange outer one (both vertex-painted with the same value-noise trick as the star cores), counter-rotating against each other so the fire layers visibly slide — plus 14 varied-size "ember" spark spheres traveling around it on independent phases with a small per-spark radial/vertical wobble so they read as thrown-off embers rather than beads on a wire. The ring's radius is HALF the stars' separation and its center sits at the midpoint between them, so the circle passes through *both* star centers by construction (a plasma bridge threading the pair, not a halo around the primary — an earlier version was a full-separation-radius ring centered on the primary, which only ever touched the orbiting star). `updateStarMesh` re-derives the midpoint from the smaller star's orbit angle every frame (primary is at the local origin, so midpoint = half the orbiter's position), yaws the ring group so its local X runs along the star-to-star axis (`rotation.order = 'YXZ'`), then applies a slow oscillating roll (`weaveSpeed`/`weaveAmplitude`) around that axis — both suns lie ON the roll axis, so the ring weaves and tilts out of the orbital plane while always still intersecting both of them exactly. Each torus lives inside its own `spinner` group whose local-Y spin flows the fire pattern around the circle — the old `rotateOnWorldAxis` approach breaks once the parent ring group itself re-orients every frame. Sparks are children of the ring group positioned on its local circle, so they ride along through every re-orientation for free.
- `main.js`'s `loadBodiesForCurrentSystem()` removes/clears any existing `starMesh`/`bodyMeshes` itself before rebuilding, rather than trusting each caller to do that first — a caller that forgot (or a future one that does) is exactly how two systems' stars/bodies could end up stacked in the same scene at once (observed as an extra-stars bug during development). Its two current callers (`clearSession`, the hyperspace jump handler) still also clear it themselves for the other per-system state (`npcMeshes`, `projectileMeshes`, `impactFlashes`) it doesn't own — that redundancy for `starMesh`/`bodyMeshes` specifically is now harmless belt-and-suspenders, not load-bearing.
- `stationMesh.js` exports both the original `buildStationMesh()` (unchanged, zero-arg, always the same ring-and-arms shape) and a new `buildStationMeshForBody(body)` that hashes `body.id` to pick one of `STATION_VARIANTS`: 4 hand-crafted archetypes (`buildStationMesh` itself, `buildSpireStation`, `buildModularStation`, `buildDomeStation`) plus 40 more generated by `buildParametricStation` — a combiner over a small core-shape pool (box/cylinder/cone/sphere/torus) and accessory pool (ring/arms/spikes/modules/stack/panels). Each of the 40 has its own fixed `structureRng` seed (so archetype *N* always has the same silhouette everywhere), while colors still vary per-instance via the body's own `colorRng` — the same hand-crafted-vs-generated split `data/shipClasses.js` uses for the ship roster. `buildStationMesh()`'s exact structure (particularly a child mesh named `'ring'`) must stay stable because `main.js`'s main-menu flyby (`updateMenuBackground`) calls `.getObjectByName('ring')` on it directly — the flyby always calls the zero-arg function, never `buildStationMeshForBody`, so it's unaffected by the new variety.
- All station archetypes (hand-crafted and generated) carry pulsing nav-beacon lights (`addBeacon`: a small emissive sphere + a bigger additive "glow" halo, same corona-shell technique as the star) and most carry a soft additive glow ring/halo (`addGlowRing`) around their most prominent circular feature. `buildStationMeshForBody` also sets a small per-body `userData.spinSpeed` on the returned group. `updateStationMesh(mesh, elapsed, dt)` — called each frame in `main.js` over every entry in `bodyMeshes` — applies that spin and pulses each beacon's glow opacity; it's a no-op for any mesh without that userData, so it's safe to call unconditionally. Since it only checks `userData.spinSpeed`/`userData.beacons` rather than anything station-specific, `buildPlanetMesh` also sets `userData.spinSpeed` on planet/moon meshes to get a free, slow axial spin out of the same per-frame loop.
- Moons visually orbit their parent planet: `main.js`'s `loadBodiesForCurrentSystem` derives each moon's orbit radius/starting angle from its already-generated position relative to its `parentId` parent (no extra persisted field), picks an orbit speed off a hash of the moon's id, and stores it in the module-level `moonOrbits` map. Every frame, `animate()` recomputes that moon's `body.position` (not just its mesh) from the orbit and writes both back — real movement of the authoritative game-state position, not a purely cosmetic mesh offset, so collision/docking/probing/radar (which all read `body.position` fresh each frame) stay consistent with what's rendered. The orbit is a flat circle at a *constant* XZ-plane radius (only the angle advances, `orbit.y` never changes), so `galaxy.js`'s `localPositionNearBody` clamps that radius at generation time to clear `parentRadius + moonRadius + MOON_ORBIT_CLEARANCE_MARGIN` — guaranteeing clearance once at generation guarantees it for the entire orbit, since the radius never shrinks afterward.
- `render/nebula.js`'s `createNebula()` scatters several large, softly-glowing additive sprite clusters (a Canvas-2D-generated radial-gradient texture) around a few random centers, so the starfield background reads as nebula/cosmic dust rather than empty black. Each puff's hue drifts slightly off its cluster's base (real emission nebulae aren't one flat color) and is stretched into a randomly-rotated ellipse (`SpriteMaterial.rotation` + non-uniform `scale`) — overlapping elongated wisps at different angles read as filamentary gas structure instead of a pile of circles. A handful of huge, extremely faint neutral-grey veils are also spread wide between the colored clusters as inter-nebula "cosmic dust" haze. Cluster centers are placed at a *minimum* radial distance from world origin (~2800+), not just anywhere in a bounding box, so a cluster never spawns close enough to loom over and wash out a system's view. Not seeded off the galaxy PRNG (same convention as `starfield.js`'s plain `Math.random()` — it's fixed decoration, not gameplay-relevant content).
- `starfield.js` renders three size tiers of stars (dense dim pinpricks, a sparser mid layer, and ~60 big "hero" glow stars) as `PointsMaterial` with a shared canvas radial-gradient sprite map + additive blending — without a map, `PointsMaterial` draws each star as a hard square pixel block, which is exactly how it used to look. ~2% of the field twinkles: those stars live in four small extra layers whose whole-material opacity oscillates on its own phase/speed (`updateStarfield`, called each frame from `main.js` right after the recentering below) — cheap per-layer uniform animation instead of a per-star shader; with the stars scattered randomly across four out-of-phase groups it still reads as individual twinkling.
- Both `starfield.js`'s star field and the nebula are plain point/sprite clouds scattered once around world origin at creation time — with every system sharing that same (0,0,0) local origin (see below) but now spanning up to ~8000 units across, the camera routinely ends up outside their fixed radius during normal play. `main.js`'s `animate()` re-centers both (`starfield.position.copy(camera.position)`, same for `nebula`) every frame, unconditionally at the very top before any of the early-return branches (menu/jump/dock/paused) — this is what actually makes them read as a skybox that always surrounds the viewer, rather than a fixed patch of decoration the camera can fly outside of. `updateNebula(nebula, dt)` additionally gives it a very slow constant rotation for a subtle "swirl", called right after that recentering.

### Collision

`game/collision.js`'s `resolveBodyCollisions` runs every frame against the player ship only (not NPCs): flying into a planet/moon/asteroidField/station/settlement pushes the ship back to the surface and cancels the inward velocity component (a slide, not a bounce or damage). Planet/moon/asteroidField use `body.radius`; station/settlement use fixed per-kind constants since their mesh size never varies. `main.js`'s dock-range check (`dockRangeFor`) has to stay wider than a body's collision shell (`collisionRadiusFor(body) + shipRadius`), or docking would be physically unreachable — this bit the first pass (station collision radius alone pushed the ship outside the old flat `DOCK_RANGE`).

### Docking animation

Docking/undocking is a scripted two-half position lerp (`main.js`'s `dockEffect` state machine, mirroring `jumpEffect`'s pattern): fly from wherever the ship is to just outside the body, swap the *entire* exterior scene (bodies/star/NPCs/projectiles) out for a shared generic bay interior (`render/stationInterior.js`) behind a brief flash — the flash (the same `jumpFlashEl` reused for hyperspace, recolored per effect) masks the cut — then glide to the parked spot. Undocking reverses the exact same path; `dockedApproach` remembers the original approach vector so the reverse trip lines up. The interior lives at a fixed coordinate far outside any real system's range (`DOCKING_BAY_ORIGIN`), so meshes are only added/removed from the scene, never rebuilt.

Repairing (`game/economy.js`'s `repairCost`/`repairShip`) lives in `ui/dockingUI.js`'s Shipyard tab (not Trade — buying/selling ships and repairing are both "keep the ship healthy" actions), and is offered at both stations and settlements (`canRepairHere` checks `body.kind`), unlike buying/selling ships which stays station-only since settlements never generate `hasShipyard: true`. It tops up hull and armor only — shields already regenerate for free on their own over time (`combat.js`'s `regenShields`), and additionally get topped up to full for free the instant the player docks (`main.js`'s `dock()`), so only hull/armor ever cost credits. Cost is `REPAIR_COST_PER_POINT` (5cr) scaled by both how damaged the ship is (missing hull+armor points) and how big it is (`shipClass.hull.length / REPAIR_SIZE_REFERENCE_LENGTH`, the reference length being `bravia_mk2`'s own, so the starter ship's rate is unchanged at exactly 5cr/point) — a flat per-point rate stopped making sense once ship sizes started varying by 2-3x. Settlements charge a further `SETTLEMENT_REPAIR_SURCHARGE` (+5%) on top, since their parts supply is more limited than a full station's.

The Shipyard tab also shows a stats panel (`ui/dockingUI.js`'s `renderShipStatsPanel`, sharing one `SHIP_STAT_ROWS` list so browsing-a-ship-for-sale and viewing-your-own-ship read identically) next to the ship-for-sale table: clicking any row (not just its Buy button) sets `selectedShipClassId` and re-renders, defaulting to (and resetting to, every time the tab reopens at a new body) the player's own current ship. Buying a ship no longer prompts for a name — a fresh purchase just takes the class's stock name — since asking for one at the moment of purchase was specifically what got removed; `game/economy.js`'s `renameActiveShip`/`renameStoredShip` (both just trim-and-validate then set `instanceName`) are exposed instead via a "Rename" button next to the current ship in the Shipyard tab and next to each stored ship in the Storage tab, so renaming only ever happens for a ship the player already owns.

### Main menu

- `ui/menu.js`'s title (`h1`) uses `drop-shadow` filters (not `box-shadow`, so the glow hugs the actual glyph shapes) animated through several hues for a slow "gaseous" halo, plus `::before`/`::after` pseudo-elements duplicating the text with opposite `clip-path` bands, `hue-rotate` filters, and `mix-blend-mode: screen` — normally invisible (`opacity: 0`), briefly flashed on by `steps(1)` keyframes for ~1 second out of every ~6.5s loop for an occasional chromatic-aberration glitch rather than constant noise. The title also plays a one-shot resolve-out-of-a-blur entrance (`titleEntrance`, hooked on the same `.reveal` class the button stagger uses, so it replays whenever the menu re-shows), sized at 46px over a 560px panel (46px over the old 420px wrapped to three lines). The screen is framed by corner braces (`.frame .corner` — the same element as the HUD's cockpit frame), darkened toward the edges by a vignette (`#main-menu::after` radial gradient), flanked-rule lines around the subtitle (its `::before`/`::after`), a fictional status footer, and a center-out glowing underline sweep on menu-link hover (the *button's* `::after` — `.glitch-text`'s own two pseudos are already taken by the glitch layers).
- The top-level New Game/Load Game/Quit entries (`button.menu-link`) are plain text, not boxed buttons — chrome (background/border/padding-box) stripped via CSS but still real `<button>` elements for click handling/disabled state. Each wraps its label in a `.glitch-text` span carrying the same `data-text`-mirrored `::before`/`::after` clip-path-band technique as the title (just tinted via `hue-rotate` on a flat color instead of a gradient, since there's no `background-clip: text` gradient to shift here), staggered per item via `animation-delay` (`nth-of-type`) so all three don't glitch in sync. Hovering (or `:focus-visible`) swaps the hover state to a stacked `drop-shadow` glow instead of a box-shadow — bright and multi-layered rather than a single soft shadow, red-tinted for Quit, blue/green for the other two.
- The background is a binary star the camera slowly orbits (`render/starMesh.js`'s `buildStarMesh({ id: ... }, 'binary')` — the second, optional `forceType` param skips the hashed random pick so it's always a binary pair, never left to luck). Real binary stars are thousands of units across post-`STAR_SIZE_SCALE`, far too big for this tiny stage — rather than a hardcoded shrink multiplier (which needed re-tuning the one time `STAR_SIZE_SCALE` changed after this was written), `main.js`'s `startMenuBackground` measures the built mesh's bounding sphere and normalizes it to a fixed on-screen radius (`MENU_STAR_DISPLAY_RADIUS`), so it self-adjusts regardless of how big the underlying game-scale stars get. The pair itself sits fixed at `MENU_STAR_CENTER`; `updateMenuBackground` instead moves the *camera* in a circle around that point (`MENU_ORBIT_RADIUS`/`MENU_ORBIT_HEIGHT`, one lap per `MENU_ORBIT_PERIOD_S`) with `camera.lookAt` tracking a point offset above `MENU_STAR_CENTER` by `MENU_LOOKAT_LIFT` (not the star itself) — since `#main-menu`'s flex centering always puts the title text dead in the middle of the screen and `camera.lookAt(target)` always puts `target` dead in the middle of the frame, looking directly at the star (as an earlier version did) put it right behind the title text regardless of the star's own world-space position; looking slightly above it instead pushes the star down in-frame, clear of the text. `updateStarMesh` (the same function used for real in-system stars) still separately drives the pair's own spin/pulse/orbit/energy-ring animation, oblivious to the camera circling it. That energy ring's tube/spark thickness (`render/starMesh.js`'s `buildEnergyRing`) is sized proportionally to the ring's own radius rather than a flat constant — the flat value predated three rounds of `STAR_SIZE_SCALE` growth and had shrunk to a barely-visible hairline by the time the ring's radius (which scales with the stars, since separation is derived from their corona reach) grew to thousands of units around it, most noticeable at the menu's heavily-normalized-down display size but present at real in-game scale too.

### Controls: mouse-aim flight mode

Flight is mouse-aim, toggled by Space (`main.js`'s `flightMode`), not always-on:

- Entering flight mode calls `renderer.domElement.requestPointerLock()`; if that promise rejects, `flightMode` is set back to `false` immediately in the `.catch()` rather than waiting on the uncertain-timing `pointerlockchange` event (a `pointerlockchange` listener still syncs `flightMode` to `false` if the lock is released some other way after a successful acquisition, e.g. the browser's own forced-unlock behavior). Exiting flight mode (`exitFlightMode()`) explicitly hides the crosshair too, since `updateCrosshair()` only runs in the per-frame gameplay branch of `animate()` — it never gets a chance to hide itself if a menu opens and that branch stops running (bit us once: crosshair stayed frozen visible behind the pause menu until `exitFlightMode` was made to hide it directly).
- Opening the pause menu, nav map, docking UI, or inventory screen all call `exitFlightMode()` so the OS pointer is available for clicking UI. Closing any of them calls the mirror `reenterFlightMode()` (requests pointer lock again, same fire-and-forget `.catch()` pattern), so the player lands back in flight controls without pressing Space themselves — this only applies to *closing* one of these overlays, not to opening the game fresh. A brand-new game (not a loaded save) additionally defaults straight into flight mode via `startSession`'s `enterFlightMode` option, requested as a direct continuation of the "Launch" button's click so it still counts as a live user gesture. Undocking requests it immediately on the Undock button click too (rather than waiting for the ~2.4s undock animation to finish), for the same reason — by the time an animation completes on its own via `requestAnimationFrame`, the click that started it is no longer a "live" gesture as far as Chromium's Pointer Lock grant is concerned.
- `game/flight.js`'s `updateFlight(shipState, shipClass, keys, mouseAim, dt)`: mouse deltas (`createMouseAimState()`, accumulated only while `document.pointerLockElement` is set, consumed/reset every call) directly rotate the ship's yaw/pitch, scaled by that ship class's `turnRate`; pitch is inverted (mouse up → nose up, the traditional flight-stick mapping). Q/E still roll on keys (a 2-axis mouse can't drive a third axis). W/S ramp a persisted `shipState.throttle` (-1 to 1) rather than applying raw per-frame thrust, so releasing both keys holds the current cruising speed instead of coasting to a stop; below zero, S applies reverse thrust, separately capped (via the forward-component of velocity, not the general speed clamp) to `REVERSE_SPEED_FRACTION` (25%) of the ship's forward max speed. A/D are direct lateral (strafe) thrust, independent of throttle.
- Firing is mouse-button-specific and gated by `flightMode`: left-click fires only `laser`-type hardpoints, right-click only `missile`-type ones, via `combat.js`'s `fireProjectile`'s optional `weaponTypeFilter` param (ships without a matching hardpoint simply fire nothing — no special-casing needed). NPC calls omit the filter and still fire every hardpoint, unchanged. `contextmenu` is prevented globally so right-click doesn't pop the OS context menu. Both keys and mouse deltas are only applied to `updateFlight` while `flightMode` is true (`flightMode ? keys : EMPTY_KEYS`); outside flight mode the ship still coasts under existing damping (physics doesn't pause), it just stops responding to input.
- The crosshair (`updateCrosshair()`) projects a fixed point ahead of the ship (`CROSSHAIR_DISTANCE`) the same way `updateWaypointIndicator()` projects the waypoint arrow — it shows the ship's actual instantaneous forward vector (where fixed hardpoints will fire), not the camera's, since the chase camera lags behind via `lerp` and can visibly diverge from the ship's facing during fast mouse turns.
- The HUD's velocity gauge (`hud.js`'s `.bar.velocity`) is signed and center-zero: it's fed the *forward component* of velocity (`velocity.dot(shipForward)`, computed in `main.js`), not the unsigned speed magnitude shown in the text readout next to it — fill extends right (green) for forward, left (amber, `.reversing` class) for reverse.
- Known gap: Pointer Lock's engagement itself could not be verified end-to-end via CDP-scripted testing — Chromium requires genuine OS-level window focus for `requestPointerLock()` to succeed, which an automated/background-launched Electron window may not have (`WrongDocumentError` observed in that environment specifically). The `flightMode` state, mouse-button-to-weapon-type routing, throttle/reverse/strafe/roll math, and menu-forces-pointer-mode behavior were all confirmed live (the weapon-routing check used a direct `flightMode` override to sidestep the broken pointer lock, since its rejection now resolves fast enough to flip `flightMode` back off within a single frame); the actual pointer-capture UX should be spot-checked by a human running the packaged app normally.

### Targeting

Tab locks a combat/scan target — ships (NPCs), stations/settlements, individual asteroids within a field, and wrecks; planets and moons are deliberately excluded (they're dock/probe candidates, never a target). State lives in `main.js`'s module-level `currentTarget`, never persisted (NPCs and wrecks aren't either): `{ kind: 'npc'|'body'|'wreck', id }` for ships/stations/wrecks, or `{ kind: 'asteroid', fieldId, index }` for a specific rock, since one asteroid field body produces many targetable rocks rather than one. `RADAR_RANGE` and `TARGET_RANGE` are both **1500** (`main.js`).

- `render/asteroidFieldMesh.js`'s `getAsteroidRocks(body)` is the single source of truth for a field's scattered rock positions (radius/position/rotation/scale, all derived from `mulberry32(hashString(body.id))`) — `buildAsteroidFieldMesh` calls it to build the visible rocks, and `main.js`'s targeting code calls the exact same function to compute the same rocks' world positions, so a targeted reticle always lines up with the rock actually rendered.
- First Tab press (or if the current target's gone — destroyed NPC, left the system) calls `cycleTarget()`'s "acquire" path: among all in-range targetable entities, it picks whichever is most aligned with the ship's actual forward vector (`forward.dot(direction) > ~0.94`, a ~20deg forgiving cone) rather than a precise raycast — i.e. "what's under the crosshair", since the crosshair itself just projects that same forward vector (`CROSSHAIR_DISTANCE` ahead).
- Subsequent presses (target still valid) instead sort all in-range entities by distance and jump to the one after the current target, wrapping around — "cycle between nearby targets". `sameTarget`/`toTargetRef` centralize the (fieldId, index) vs. plain id comparison so this logic doesn't need to special-case asteroids inline.
- `resolveTarget()` looks the target up fresh every frame (never cached), so a destroyed NPC correctly resolves to `null` next frame rather than holding a stale position; it also returns `isAsteroid`, which both `updateTargetIndicator` (reticle tint) and `updateMiningBeam` (auto-fire eligibility — see Mining mode) key off of.
- `updateTargetIndicator()` projects the resolved target's position the same way `updateCrosshair`/`updateWaypointIndicator` do, and shows a bracket + name/distance/hull% label (hull% only for NPCs) tinted red for hostiles (`isHostileToPlayer`, see Combat AI below), amber for asteroids, or neutral blue otherwise; hidden when the target is behind the camera. Hidden explicitly in `exitFlightMode()` too, the same fix already needed for the crosshair (see above) — a per-frame update that only runs in the gameplay branch of `animate()` can't hide itself once a menu stops that branch from running.
- Hardpoints often sit off the centerline (especially +Y). `combat.js`'s `fireProjectile` aims each shot at a boresight convergence point (`AIM_CONVERGENCE`, ~350m ahead on ship forward) rather than flying parallel to forward from the mount — otherwise raised guns land above the crosshair.

### Missions UI & map markers

- `ui/missionsUI.js` (toggled by **J**, same exit/reenter-flight-mode pattern as inventory/nav map) lists active missions with type, reward, current nav target, and progress/ready-to-turn-in status; **Set Waypoint** / **Waypoint: Turn-In** calls `setWaypointForMission`.
- `game/missions.js`'s `missionNavTarget(mission, gameState)` is the single source of "where should the player go next": incomplete → objective system/body (or bounty position/NPC), complete → giver station (`giverSystemId`/`giverStationId`). `missionMarkedSystemIds` / `missionMarkedBodyIds` feed the galaxy map and system list.
- Galaxy map (`navMap.js`): orange ring under systems returned by `missionMarkedSystemIds`. System tab: mission bodies get a `mission-marker` row + "mission" tag; setting a body waypoint clears `waypointPosition`.
- Radar contacts use kind `mission` (orange) for marked bodies and free-space bounty waypoints; tracked mission waypoints use an orange arrow in `updateWaypointIndicator`.

### Combat AI: factions, pirate difficulty, and the pirate/alien truce

- NPC factions are `pirate`, `alien`, or `trader`. Traders never fight (`opponentsFor` in `combat.js` returns no opponents for them). Aliens are always hostile to both the player and pirates. Pirates are hostile to the player *unless* a truce is in effect (see below), in which case they target aliens instead.
- `combat.js`'s `opponentsFor(npc, gameState)` is the single place that decides who an NPC is currently fighting — `updateNpcAI` picks the nearest entry from it (instead of always hardcoding the player) to drive the existing patrol/attack/flee state machine, so the same state machine works for NPC-vs-player and NPC-vs-NPC without duplicating it.
- `fireProjectile`'s optional `targetRef` (`{ kind: 'player' }` or `{ kind: 'npc', id }`) records what an NPC's shot is actually aimed at, since `updateProjectiles` can no longer assume every non-player projectile is headed at the player — the player's own shots omit it and keep hitting any NPC in range, unchanged.
- `combat.js`'s `truceActive(gameState)` is simply "is there a live alien in this system" — it doesn't track discrete "arrival" events, since a pirate/alien encounter already forming, an alien arriving mid-pirate-fight, and a pirate arriving mid-alien-fight all reduce to the same continuous condition once phrased this way. `updateCombatFlag` (the no-save-during-combat gate) and `main.js`'s `isHostileToPlayer` (radar dot color, target reticle tint) both key off the same truce state so a truced pirate doesn't read as hostile to the player.
- The player's starting system (`gameState.player.startingSystemId`, distinct from `currentSystemId` which changes as the player travels) stays free of *hostile* ambient encounters until the player breaks that peace themselves by firing on a non-hostile ship (a trader) while actually there — `combat.js`'s `updateProjectiles` sets `gameState.flags.startingSystemPeaceBroken = true` the moment that happens, checked against `currentSystemId === startingSystemId` so shooting a trader elsewhere doesn't affect it. Until broken, `main.js`'s ambient-spawn loop passes `forceNeutral: true` into `spawnEncounterNear` (`game/spawner.js`), which skips the pirate/alien rolls entirely and always spawns a trader — so ambient traffic still occurs at home, it's just never hostile. This is layered on top of (not a replacement for) the existing core/rim population and alien-activity gradient below; a mission-driven bounty NPC can still spawn in the starting system regardless (deliberately accepted, not an ambient encounter).
- `game/spawner.js`'s `pickPirateShipClass` sizes pirate difficulty off `coreFraction` (the same value driving the population/alien gradient — see Procedural generation): it draws from a band of the ship roster sorted by price, sliding from the cheapest/weakest third near the galactic core to the priciest/toughest third at the rim, so pirates get harder to fight the further out you go.
- `main.js`'s per-frame loop edge-detects the truce ending (`truceWasActive` was true, `truceActive` is now false — i.e. the aliens are all dead or gone) and has any surviving pirates leave: it shows a toast ("The pirates thank you...") and removes them from `gameState.npcs` directly, rather than marking them `destroyed`, so a truced pirate that happens to be an active bounty target doesn't auto-complete that mission — it just re-materializes next time the player re-enters the system (the same existing respawn path `ensureBountyNpcsForSystem` already uses).
- `updateNpcAI`'s `attack` state has two variations layered on top of the base "face and close on the opponent" behavior, both driven by per-NPC flags lazily set once (`??=`) rather than re-rolled every frame: `isOrbiter` (~50% of attackers) circles the target once within `ORBIT_RANGE` instead of always closing head-on — thrust goes mostly tangential while facing (and firing) still tracks the opponent directly, via a perpendicular vector (`(-toOpponent.z, 0, toOpponent.x)`) scaled by a per-NPC `orbitDir` (+1/-1) picked once for a consistent circling direction. Independently, any NPC hit within the last `EVADE_WINDOW_S` (checked off the existing `lastHitAt`, no new field) thrusts backward instead of closing in — a brief break-off after taking a hit, still facing (and still able to fire at) the opponent while it retreats.
- The moment an NPC's hull first drops below `FLEE_HULL_FRACTION`, it rolls once (not every frame — see the comment in `combat.js`) between the existing `flee` state and a new `ram` state (`RAM_CHANCE`, ~3%): a suicide run aimed squarely at the *player* specifically (not `opponentsFor`'s pick — a truced pirate's opponent is an alien, but a ram is always about the player), turning/accelerating faster than normal and dealing `RAM_DAMAGE` plus destroying itself on contact. `main.js`'s NPC loop shows one of `RAM_QUOTES` ("Ramming speed!", etc.) via the faction toast (recolored red for this, vs. its green default for the pirate-truce message) the first frame `aiState === 'ram'`, guarded by a one-shot `npc.ramAnnounced` flag.

### Thruster particles

`render/thrusterParticles.js`'s `createThrusterEffects()` builds three independent emitters sharing one group, added to the scene in `startSession` and updated every frame alongside `syncMeshToEntity(playerMesh, ...)` in `main.js`: a warm rear exhaust puff while accelerating (W), a smaller cool-toned puff at the nose while braking (S), and elongated cyan streaks trailing from the rear during supercruise. All three key off the ship's hull length (`hullLength / 2` for the front/rear Z offsets — the hull is centered at the local origin, per `procgen/hull.js`) so they always emit from the actual nose/tail regardless of ship class.

- The accel/brake puffs (`createPuffEmitter`) are round additive glow sprites (the same canvas-radial-gradient technique as `render/nebula.js`) recycled round-robin in a fixed-size pool. There's deliberately no per-particle fade/lifetime: each just keeps drifting in a straight line at its own spawn velocity forever, so older particles end up far enough behind (or ahead of) the ship that the chase camera naturally leaves them out of frame before that pool slot needs reusing — cheaper than tracking per-particle age/opacity via a custom shader.
- The supercruise streaks (`createStreakEmitter`) are a deliberately different technique — elongated cylinders, not round points — reusing the mining beam's "orient a stretched unit shape" trick: the cylinder's length axis is baked onto local Z once at creation (`geometry.rotateX(Math.PI / 2)`), so aligning a streak with the ship's current facing each frame is just copying its quaternion directly, no per-particle direction math needed.
- Each emitter's `mesh.visible` is tied directly to its own active flag (`accelActive`/`brakeActive`/`cruiseActive`, mutually exclusive by construction since `main.js` only sets `thrustState` when `!cruising`) — no fade-out on release, particles simply vanish, which reads fine for a snappy arcade feel.

### Mining mode

`R` toggles `main.js`'s module-level `miningMode`, which swaps the laser's behavior entirely rather than adding a new input: with it on, the laser ignores `laserFireHeld`/crosshair aim and instead auto-fires a continuous beam at the current target for as long as it's an asteroid, in range, and the mining hold isn't full; with it off, lasers fire exactly as before (manual left-click, toward the crosshair). Missiles are unaffected either way.

- `updateMiningBeam()` doesn't spawn physical projectiles — there's nothing for a beam to travel toward or collide with; instead it directly ticks `mineRock` (against the specific targeted rock, `currentTarget.fieldId`/`.index`) on a timer (`MINING_TICK_INTERVAL_S`) whenever `miningMode && flightMode && !cruising && target.isAsteroid && inRange && !isMiningHoldFull()`, reusing the existing `onProjectileHit` toast/sound/flash feedback path used by manual mining hits.
- It stops on its own once the hold is full simply by never activating in that state (checked via `isMiningHoldFull()` *before* ticking), rather than firing a doomed tick and spamming the "hold full" toast every interval.
- The beam mesh (`buildMiningBeamMesh`) is a constant unit-length cylinder rescaled/reoriented between the ship's first laser hardpoint and the target's position every active frame — the same stretch-a-unit-shape technique the binary stars' old energy-trail beam used (see above) — so "no need to point at the asteroid" is literal: the beam bends to the target regardless of the ship's actual facing.
- A small `MINING MODE` HUD indicator (top-left, independent of the top-center supercruise indicator since the two toggles are unrelated and can both be true at once) gives a persistent visual reminder the mode is active, since the beam itself is only visible when it's actually firing.
- `updateMiningBeam()` also drives `audio.setMiningBeamActive()` alongside `miningBeamMesh.visible` (same condition, same call sites) — a continuous warbling hum distinct from `playLaser`'s one-shot zap, for as long as the beam is actually active (see Audio below).

### HUD layout, galaxy map, and other UI polish

- `ui/hud.js`'s three pieces are positioned independently rather than as one block: `.status-panel` (shield/armor/hull) top-left, `#radar` top-right, `.velocity-panel` bottom-center, and `.hint` (keybind chips) bottom-left — the last one deliberately unmoved from where the whole HUD used to sit, per a later layout request that repositioned only the other two. All three still live inside one `#hud` wrapper div so `hud.element.remove()` in `clearSession()` still tears down all of them in one call. The wrapper also carries two decoration-only layers (`.scanlines`, a faint full-screen repeating-gradient wash, and `.cockpit-frame`, four corner braces) for a looking-through-a-canopy read, both `pointer-events: none`; every `.bar` gets segment tick lines via a `::after` repeating-gradient overlay (the sci-fi cell-battery look — cells appear/disappear as the fill crosses each tick, no JS); and `updateRadar` draws bearing tick marks around the radar rim every 15deg (longer every 45). The main menu's `.frame .corner` braces are the same visual element, so menu and gameplay read as one system.
- `ui/navMap.js`'s galaxy-map canvas got bigger (680px, from 480) with a radial gradient glow centered on the canvas (bright core, fading toward the rim) layered under the dots — the underlying system positions already form a spiral-arm distribution (`procgen/galaxy.js`'s `spiralPosition`), so the glow alone is enough to sell "galaxy" without needing to duplicate the arm-twist formula for a guide overlay. A `mousemove` listener finds the nearest system the same way the existing click handler does (factored into a shared `nearestSystemAt`) and shows a floating tooltip with its name (plus " · mission" when that system is mission-marked), plus highlights the hovered dot. Orange rings mark systems with an active mission objective or turn-in (see Missions UI above). The Current System tab's body list gets a small inline-SVG icon per kind (`BODY_ICONS` — filled circle for planets, smaller dim circle for moons, square for stations, dimmer square for settlements, a scattered-dot cluster for asteroid fields) next to each name, cheaper and more literal than emoji.
- `ui/dockingUI.js`'s docked-location header only appends `"(kind)"` if the body's own generated name doesn't already end with that word — station/settlement names already end in "Station"/"Settlement" (`procgen/names.js`), so appending the kind again read as "Dunell Settlement (settlement)". Planets/moons don't embed their kind in the name, so they still get the suffix.
- `ui/dockingUI.js`, `ui/navMap.js`, `ui/pauseMenu.js`, and `ui/inventoryUI.js` originally each hand-rolled their own flat, undecorated look (`#0b1020` panels, `#2a3a55` borders, solid-color buttons, no glow) — a visibly plainer style than `hud.js`/`menu.js`'s glass-panel/glow/clip-path treatment, since those four were built earlier and never revisited. Brought up to match: gradient panel background + accent-color left border + outer glow + a clipped bottom-right corner (the same shape as `hud.js`'s `.status-panel`), an underlined-tab style (transparent background, glowing bottom border on `.active`) replacing the old solid-fill tabs, and semi-transparent bordered buttons that light up with a box-shadow glow on hover, color-coded by intent (cyan neutral, green for jump/repair/use-part, red for close/quit). This was a pure CSS change — every file's markup, class names, and event wiring are untouched, only the `STYLE` template string in each changed — since the inconsistency was visual, not structural. `ui/deathScreen.js` was deliberately left alone; its red-tinted glitch look is an intentional contrast with the rest of the UI's blue theme, not an oversight.
- `render/shipMesh.js`'s `addHullDetails` layers a small cockpit canopy (a flattened hemisphere near the nose — local +Z), one or two glowing engine discs at the tail (two for hulls wider than ~8% of their own length, a twin-engine silhouette), and a handful of small dark "greeble" panel blocks scattered across the spine, all seeded off `mulberry32(hashString(shipClass.id))` — per-*class*, not per-instance or `Math.random()`, so every ship of a given class always looks identical, matching the rest of the game's cosmetic-variety convention (station archetypes, planet archetypes, etc.).
- `audio.js`'s `announce(text)` wraps `window.speechSynthesis`/`SpeechSynthesisUtterance` for voice callouts ("Hyperdrive engaged/disengaged", "Supercruise engaged/disengaged") — a no-op wherever the Web Speech API isn't available rather than throwing, since it's a nice-to-have layered on the existing synthesized SFX, not a required system. Supercruise's callouts are edge-detected once per frame in `main.js`'s `animate()` (`cruising !== wasCruising`) rather than at each of the several places `cruising` can flip (the `KeyC` handler, auto-arrival, a combat interrupt), so every trigger source is covered by one check. `playDock`/`playUndock` were also both layered with a hard mechanical clunk (driven square-wave thump + noise crack) and a hydraulic hiss on top of the original tone, via a new optional `delay` param on `tone()`/`noiseBurst()` (offsets the Web Audio scheduling time, defaulting to 0) so the hiss and clunk can be genuinely sequenced rather than firing simultaneously.
- Exploration/investigation/probe missions whose target is in a different system than where they're picked up are capped to within `MAX_MISSION_JUMP_DISTANCE` (4) hyperspace jumps via `procgen/galaxy.js`'s `systemsWithinJumps` (a BFS over `neighborIds`, the same lanes `canJumpTo` enforces) — `data/missionTemplates.js`'s `pickTargetSystem` picks from that reachable set instead of the whole galaxy. Bounty missions are unaffected (their target always spawns in the giver's own system already).

### Audio

`audio.js` mixes three techniques: (1) **sample SFX** under `public/audio/sfx/` (Kenney Sci-Fi Sounds, CC0 — thrusters, brake, supercruise, lasers, rockets, dock/undock) decoded into Web Audio buffers on first user gesture (`ensureSfx` / `playSample`); (2) short **synthesized** one-shots still via `tone()`/`noiseBurst()` for hits, explosions, hyperspace, UI clicks, and as fallbacks if a sample hasn't loaded; (3) **music** as MP3 files under `public/audio/` via plain `<audio>` elements.

- Weapon fire: `playWeaponFire(weaponId)` maps each catalog id to a sample with light rate jitter; synth profiles remain as fallback. Engine loops: `setThrustState('accel'|'brake'|null)` and `setSupercruiseActive` loop sample beds with fade in/out; engage/disengage play `engine_engage.ogg`. Dock/undock: layered clamp + door + seal samples.
- `announce(text)` uses the Web Speech API with a preferred **female** English voice, slower rate (~0.72), and a soft multi-tap delay "reverb bloom" under the phrase (OS TTS can't be wet-routed into Web Audio). Supercruise callouts are edge-detected once per frame in `animate()` (`cruising !== wasCruising`).
- Music lifecycle: `playTitleMusic()`/`stopTitleMusic()` bookend the menu background; `startAmbientMusic()` cycles `bg1.mp3`–`bg6.mp3`; `playDeathMusic()` uses `ded.mp3`. Relative paths only (`audio/...`) so packaged `file://` builds resolve correctly. `resumeAudioOnGesture` unblocks autoplay and kicks off SFX load.
- Mining beam remains a continuous synthesized warble (`setMiningBeamActive`). `exitFlightMode` force-stops it because the gameplay loop early-returns when menus are open.
- `playHyperspaceArrival()` mirrors `playHyperspace()` for jump completion.

### Save/load & permadeath

- NPCs, projectiles, and wrecks are never persisted — encounters are ephemeral and re-spawn procedurally on load or hyperspace jump (`missions.js`'s `ensureBountyNpcsForSystem` re-materializes any active bounty's target); wrecks simply start empty on load, same as npcs/projectiles. `stationStorage` (cargo/ore/ship parts/ships left at a station) *is* persisted — it's tied to the galaxy's bodies, not a transient encounter.
- Saving throws if `gameState.inCombat` is true (`game/save.js`).
- On hull ≤ 0, `handlePlayerDeath()` in `main.js` ends the session and shows the death screen; it does not delete the save file.

### Per-station storage & ship parts

- `gameState.stationStorage` is keyed by body id: `{ cargo, miningHold, shipParts, ships }`, created lazily by `game/economy.js`'s `storageFor` on first use. Nothing here ever crosses to a different station — the whole point is "you can only pick up what you left behind at *this* body" — so every storage function takes a `bodyId` and only ever touches that one entry.
- Buying a ship (`purchaseShip(gameState, bodyId, classId, instanceName)`) never swaps the player's active ship — it always pushes the new ship straight into that station's `storage.ships`. The only way a stored ship becomes active is `activateStoredShip(gameState, bodyId, index)`, which swaps it with whatever's currently flown (the old active ship takes the stored ship's old slot) — so a purchase-then-activate and a swap-between-two-owned-ships are the same code path. `sellStoredShip` removes a stored (never the active) ship permanently for `STORED_SHIP_RESALE_FRACTION` (50%) of its list price.
- Cargo/ore/ship-parts transfers (`storeCargo`/`retrieveCargo`, `storeOre`/`retrieveOre`, `storeShipParts`/`retrieveShipParts`) are whole-hold, not per-good — simplest useful shape for "leave stuff behind, grab it later" without a quantity picker per good. `retrieveCargo`/`retrieveOre` throw if the combined total wouldn't fit the ship's capacity, same failure mode as a regular purchase.
- Ship parts (`data/goods.js`'s `SHIP_PARTS_GOOD_ID`) are a rare consumable, not a cargo-hold item — held as a plain count on `ship.shipParts`. Only `hasShipParts` bodies stock them (`procgen/galaxy.js`'s `makeBody`, a 6% roll for stations/settlements, same convention as `hasShipyard`); `ui/dockingUI.js`'s Trade tab only shows the buy row when `currentBody.hasShipParts`. `useShipPart` (usable anywhere, not just docked — see `ui/inventoryUI.js`) heals a flat 10% of max hull *and* max armor per part, consuming one.
- `ui/inventoryUI.js` (toggled by `I`, same exit/reenter-flight-mode pattern as the pause menu/nav map) is a read-only view of the ship's cargo/mining hold plus the "use a ship part" action, plus a "Remote Assets" list summarizing every body with anything in `stationStorage` — what's there and how much, but no retrieval button, since retrieval only makes sense while actually docked at that specific body (see `ui/dockingUI.js`'s Storage tab, which has the real transfer buttons). `ui/missionsUI.js` uses the same overlay pattern (**J**).
- Flight mode tracks **intent** separately from pointer lock (`flightModeWanted` vs `flightMode`): alt-tab drops the lock and clears stuck keys, but focus / canvas click / Space re-requests pointer lock without losing the player's desired flight state.

### Weapons & the armory

- `data/weapons.js`'s `WEAPONS` catalog is the only source of a weapon's real stats (`damage`/`speed`/`cooldownS`/`ttl`/`price`/`color`) — a ship class's `hardpoints` only say what *category* of weapon fits where (`hp.type`, `'laser'` or `'missile'`, the hardpoint's fixed mount), never the weapon itself. `BASE_WEAPON_ID` (`pulse_laser`/`rocket_pod`) is the free weapon every hardpoint starts equipped with; its stats deliberately match the original fixed presets `combat.js` used before weapons became swappable, so an untouched loadout still plays identically to before this existed.
- The actual equipped weapon per hardpoint lives on ship state, not the class: `ship.equippedWeapons` (`{ [hardpointId]: weaponId }`) — set to `defaultLoadoutFor(shipClass)` for the starter ship, a freshly purchased ship, and as a load-time fallback for saves written before this existed. `combat.js`'s `fireProjectile` looks up `shooter.equippedWeapons[hp.id]` (falling back to that mount's base weapon if unset — the path every NPC takes, since NPCs never shop) to get the damage/speed/cooldown/ttl for that shot, rather than a fixed per-`hp.type` preset.
- Weapons are bought into a station's storage (`game/economy.js`'s `buyWeapon`), never straight onto the ship — same "purchase lands in storage, not equipped" shape `purchaseShip` already uses for ships. `equipWeapon(gameState, bodyId, hardpointId, weaponId)` is the only way one actually gets flown: it requires a spare copy of that weapon sitting in *this* station's storage, checks it's the right category for the hardpoint, and swaps it with whatever was equipped there — the old weapon goes back into that same storage slot, never destroyed, mirroring `activateStoredShip`'s ship-swap pattern. `sellStoredWeapon` is the only way a stored weapon actually leaves the game, for `WEAPON_RESALE_FRACTION` (50%) of its list price. `storageFor`'s per-station storage object patches a `.weapons` map onto itself lazily (`??=`) rather than as part of its initial shape, since it was added after storage entries already existed for some players.
- `ui/dockingUI.js`'s Shipyard tab gates the Armory (buy) and Loadout (a `<select>` per hardpoint, options disabled unless that weapon is either already equipped or has a spare in storage) behind `currentBody.hasShipyard`, same as buying/selling ships — a settlement's repair bay doesn't stock hardpoint weapons either. The Storage tab lists any spare weapons with a Sell button, the same shape as its ship parts/cargo sections.
- `render/projectileMesh.js`'s `buildProjectileMesh(weaponId, mountType)` sizes and colors a projectile straight off that weapon's catalog entry — a Plasma Cannon bolt is visibly bigger than a stock Pulse Laser, not just a recolor of one fixed laser/missile shape. `audio.js`'s `playWeaponFire(weaponId)` plays per-weapon sample SFX (with synth fallback), not one fixed laser zap / missile whoosh for everything.

### Wrecks & salvage

- `game/wrecks.js`: destroying an NPC with the *player's own* projectile (checked in `combat.js`'s `updateProjectiles` — an NPC-vs-NPC kill or a suicide ram doesn't count) spawns a wreck at the impact point via `spawnWreck`, pushed onto `gameState.wrecks`. Loot (a small stack of one random ordinary trade good, plus a `SHIP_PART_DROP_CHANCE` (15%) chance at one ship part) is rolled once at spawn time, not re-rolled per loot attempt.
- Wrecks are targetable (Tab) and lootable via `F` — `main.js`'s `KeyF` handler tries `findNearbyDockableBody()` first and only falls back to `findNearbyWreck()`/loot if nothing dockable is in range, so the same key does both without conflict. `lootWreck` adds as much as fits the cargo hold (ship parts have no cap) and removes the wreck immediately — no partial-loot-stays-behind state.
- `pruneWrecks(gameState)`, called once per frame from `main.js`, removes any wreck older than `WRECK_DESPAWN_S` (1800 = ~30 real-time minutes of accumulated `simTime`) — "left behind long enough, it's gone," independent of whether the player ever revisits.
- `render/wreckMesh.js`'s `buildWreckMesh()` is a small cluster of dark jagged icosahedron chunks (the same cheap-debris technique `asteroidFieldMesh.js` uses for rocks) plus a couple of pulsing ember spheres for a "still smoldering" read — not seeded off anything persisted, since wrecks themselves aren't persisted either.

### Damage effects & shield regen

- `render/damageEffects.js`'s `createDamageEffects()` reuses `thrusterParticles.js`'s round glow-sprite puff emitter (now exported) for three effects sharing one group: grey smoke once armor drops below max, orange flame plus dark scattering "debris" chunks once hull drops below max. Spawn rate/particle speed scale continuously with `1 - armorFraction`/`1 - hullFraction` (not a one-shot trigger), and each effect's origin is rejittered to a random point across the hull's own volume every frame so emission wanders rather than pouring from one fixed spot. Because the emitters key off the ship's *live* armor/hull fraction every frame, the whole effect vanishes the instant a repair (station shipyard or a ship part) brings that fraction back to 1 — no separate "clear on repair" logic needed.
- Shields already regenerated for free over time before this (`combat.js`'s `regenShields`, gated by `SHIELD_REGEN_DELAY_S`/`SHIELD_REGEN_RATE` so it's not instant) — `main.js`'s `dock()` additionally tops shields to full immediately and for free the moment the player docks, so only hull/armor ever cost credits at the shipyard (see Docking animation above).

## Testing conventions

Tests use `node:test` + `node:assert/strict` only, colocated as `*.test.js` beside the module under test — no framework, no config, no mocking layer. Keep new tests in that style.

# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Whispers In The Void — a procedurally generated 3D space trading/combat/exploration game built with Electron + Three.js (WebGL), no UI framework (plain DOM overlays for HUD/menus/trading). Real-time arcade flight/combat, one seeded galaxy of ~1500 planets and 100 ship classes, permadeath.

## Commands

```
npm run dev       # electron-vite dev — launches the app with HMR
npm run build     # electron-vite build
npm run package   # build + electron-builder --dir (unpacked)
npm run make      # build + electron-builder (installer for the current platform)
```

There is no `npm test` script. Tests use Node's built-in test runner directly, colocated as `*.test.js` next to the module they cover:

```
node --test src/renderer/game/*.test.js src/renderer/procgen/*.test.js src/renderer/data/*.test.js   # everything
node --test src/renderer/game/combat.test.js                              # single file
```

Editing `src/renderer/main.js` while `npm run dev` is running triggers HMR, which resets the running game back to the main menu — expected, not a bug.

To drive the app programmatically (e.g. for scripted verification), launch with `DEV_REMOTE_DEBUG=<port> npm run dev`; this appends `--remote-debugging-port` in `src/main/index.js` so it can be attached to over the Chrome DevTools Protocol.

## Architecture

### Process split

- `src/main` — `index.js` creates the `BrowserWindow`; `ipc.js` registers `save-game`/`load-game`/`delete-save`/`has-save`/`quit-app` handlers backed by `save.js` (plain `fs` read/write of JSON to `app.getPath('userData')/saves/save1.json`).
- `src/preload` — `contextBridge` exposes only `saveGame`/`loadGame`/`deleteSave`/`hasSave`/`quitApp` as `window.electronAPI` (`contextIsolation: true`, `nodeIntegration: false`).
- `src/renderer` — the entire game: Three.js scene plus DOM overlays. Everything below is under here.

### Renderer layout

- `procgen/` — pure, seeded generation: `prng.js` (`mulberry32` + `pick`/`range`/`intRange` helpers), `galaxy.js` (systems and their bodies), `names.js`, `hull.js` (parametric ship hull geometry — cross-section "stations" lofted into a `BufferGeometry`, plus optional wings), `shipRoster.js` (procedurally generates the bulk of the ship classes).
- `data/` — mostly static tables: `shipClasses.js` (7 hand-crafted classes in `HAND_CRAFTED_SHIP_CLASSES` + 43 more from `generateShipClassRoster`, seeded, for 50 total), `goods.js` (includes `MINED_ORE_GOOD_IDS`, kept out of the regular cargo trade table — see mining below), `economyTags.js`, `missionTemplates.js` (one generator function per mission type, including `probe`).
- `game/` — pure gameplay logic operating on plain data via a `gameState` object (no direct Three.js scene/mesh references, only `THREE.Vector3`/`Quaternion` math): `state.js`, `flight.js`, `combat.js`, `economy.js`, `missions.js`, `hyperspace.js`, `supercruise.js`, `spawner.js`, `probe.js` (random survey-data find on probing a body), `mining.js` (mining an asteroid field for ore — triggered by `combat.js` when a player projectile hits an `asteroidField` body, not a dedicated key), `collision.js` (`resolveBodyCollisions`: sphere-sphere push-out + inward-velocity cancel against system bodies — no damage, just deflection), `save.js` (serialize/deserialize `gameState`, calls `window.electronAPI`).
- `render/` — Three.js mesh builders and scene/camera helpers: `scene.js`, `starfield.js`, `shipMesh.js`, `stationMesh.js`, `stationInterior.js` (generic docking-bay interior shown behind the docking UI), `planetMesh.js`, `starMesh.js`, `asteroidFieldMesh.js`, `projectileMesh.js`, `sceneSync.js`.
- `ui/` — DOM overlays: `menu.js`, `hud.js`, `dockingUI.js`, `navMap.js`, `pauseMenu.js`, `deathScreen.js`.
- `main.js` — the orchestrator. Owns the `animate()` frame loop, wires `game/` logic to `render/` meshes and `ui/` overlays, and owns session lifecycle (`startSession`/`clearSession`/`returnToMenu`), docking eligibility, hyperspace/supercruise effect state machines, and radar contact computation.

### `gameState`

A single mutable object created by `game/state.js` and passed as the first argument through nearly every `game/` function. Holds `player` (ship, credits, reputation, `currentSystemId`, `waypointBodyId`), `galaxy` (systems/bodies from procgen), `missions` (`available`/`active`), `npcs`, `projectiles`, `economyOverrides`, `visitedBodyIds`, `probedBodyIds` (distinct from `visitedBodyIds` — set only when a body is actually probed via `game/probe.js`, not merely flown near or docked at; this is what lets `probe`-type missions require an actual probe action), `inCombat`, `simTime`, `flags`. The player's `ship` has two separate hold pools — `cargo` (regular trade goods, capacity `cargoCapacity`) and `miningHold` (mined ore only, capacity `miningCapacity`) — kept apart so mining never interacts with regular cargo trading. `game/save.js`'s `serializeGameState` defines exactly what gets persisted — `npcs` and `projectiles` are deliberately excluded (see Save/load below).

### Timing

All cooldowns, shield regen, and combat/AI timers are driven by `gameState.simTime` (seconds accumulated from per-frame `dt` in `main.js`'s `animate()`), never wall-clock time. This keeps combat logic deterministic and testable independent of frame rate — see `combat.js`.

### Ship/camera coordinate convention

This is non-obvious and spans several files — read this before touching flight, combat AI, or the camera:

- Ships treat local **+Z as forward**, +X as right, +Y as up (right-handed: right×up=forward). Used throughout `flight.js`, `combat.js`, `hyperspace.js`, `supercruise.js`, and hardpoint position data.
- `THREE.Matrix4.lookAt(target, source, up)` is deliberately called with the eye/target arguments swapped in `combat.js` (NPC facing) and `supercruise.js` (autopilot facing), because Three.js's `lookAt` follows the camera convention (local +Z points *away* from the target) while ships use local +Z as forward.
- The chase camera (`render/sceneSync.js`'s `syncChaseCamera`) sits behind and above the ship via `camera.lookAt()` with a world-up vector. Because camera space is -Z-forward (the opposite of the ship's +Z-forward), the camera's actual world-space "right" ends up equal to the ship's **local -X**. This is why `main.js`'s `computeRadarContacts()` negates `rel.x` — the comment there has the full derivation.

### Procedural generation

- A single `mulberry32(seed)` PRNG drives all galaxy generation (system layout, bodies, tags, names, missions) so a seed reproduces the same galaxy. Save files store the fully-generated galaxy array directly rather than replaying the RNG.
- Visuals/physics that don't need their own persisted field (star size/hue, asteroid rock scatter within a field) are derived from a deterministic `hashString(body.id)` hash instead. Planet/moon/asteroidField size is the exception: it's rolled once at generation time and stored as `body.radius`, since it's both a render size and a collision size (see `game/collision.js`) and needs to agree between the two.
- Body kinds: `planet`, `moon` (orbits close to one specific parent planet, referenced by `parentId`; ~23% of planets have one), `station`, `settlement`, `asteroidField` (scattered across a small fraction of systems). Only `station`/`settlement` are always dockable; a `planet`/`moon` is dockable only if it has a base, which reuses the existing `hasMissions` flag as that signal (see `isDockable` in `main.js`). `asteroidField` is never dockable, but is mineable (see below); `planet`/`moon`/`asteroidField` are all probeable regardless of whether they have a base.
- Galaxy scale is controlled by `generateGalaxy`'s default `opts` in `galaxy.js`: 450 systems, 1500 planets, 180 stations, 120 settlements, 40 asteroid fields. Ship class count is controlled the same way in `data/shipClasses.js` (7 hand-crafted + a generated count passed to `generateShipClassRoster`, currently 93, for 100 total).
- Every ship class has a `stats.miningCapacity` (mining hold size), sized off price and role via `computeMiningCapacity` in `procgen/shipRoster.js` — the starter ship (`bravia_mk2`) is hard-set below that formula's floor in `data/shipClasses.js` so it's always the smallest hold in the game.
- Mining an asteroid field (`game/mining.js`'s `mineAsteroidField`) yields one of four ore tiers (`MINED_ORE_GOOD_IDS` in `data/goods.js`) chosen by the *system's* distance from the galaxy's core (`coreFraction(system)` in `procgen/galaxy.js`, 0 at the core / 1 at the rim) — systems near the rim yield rarer, more valuable ore. Triggered by firing at an asteroid field (see `combat.js`'s `updateProjectiles`), not a separate action.
- `procgen/galaxy.js`'s `radiusFor`/`localPosition` apply `PLANET_SIZE_SCALE` (2.5x), `MOON_SIZE_SCALE` (1.65x), and `SYSTEM_SIZE_SCALE` (5x, applied to `SYSTEM_LOCAL_MIN/MAX_RADIUS`) on top of the base rolled ranges; stations are scaled 1.5x (`STATION_SCALE` in `main.js`, plus a small per-body +/-15% render-only variance from the existing `hashStringForOrbit` hash) and suns 2.5x (`STAR_SIZE_SCALE` in `render/starMesh.js`). A per-system `sizeScale` (`SYSTEM_SCALE_VARIANCE`, 0.85-1.15) is rolled once per system in `generateGalaxy` and threaded into every body placed in it (including stations/settlements/asteroid fields added after the fact, via `system.sizeScale`), so systems vary in overall size rather than all reading as uniform. `game/collision.js`'s fixed `STATION_COLLISION_RADIUS`/`SETTLEMENT_COLLISION_RADIUS` are scaled 1.5x to match (the per-body render variance isn't mirrored there — `main.js`'s `dockRangeFor` margin absorbs that small a mismatch).
- `coreFraction(system)` (`procgen/galaxy.js`) is the shared 0-1 "how far toward the rim" helper — besides ore tier, `game/spawner.js`'s `spawnEncounterNear` takes it as `alienBias` to zero out alien encounters at the core and ramp them up toward the rim (pirate odds stay fixed; trader share fills the rest), and `main.js`'s ambient-spawn loop uses it to raise the NPC cap and shorten the respawn interval near the core (busier) and do the opposite toward the rim (quieter) — implementing the "more populated near the core, more alien activity toward the rim" gradient. `game/state.js`'s `pickStartingSystem` uses the same helper to start new games in one of the 10% of systems closest to the core, rather than an arbitrary one.
- Hyperspace/new-game arrival (`SYSTEM_ARRIVAL_POSITION` in `galaxy.js`, used by both `game/state.js` and `game/hyperspace.js`) sits near the (now much larger) system's edge on -Z, scaled off `SYSTEM_LOCAL_MAX_RADIUS`. `hyperspaceJump` additionally resets the ship's quaternion to identity on arrival — since local +Z is forward and the arrival point is on -Z, this always faces the star at the system origin regardless of which way the ship was facing before the jump (new-game start already defaults to identity, so it was already correct there).

### Body/star visuals

- `render/planetMesh.js`, `render/starMesh.js`, and `render/stationMesh.js` all use a cheap seeded "value noise" — `Math.sin(nx*3.7+offsets[0]) * Math.cos(ny*4.1+offsets[1]) * Math.sin(nz*3.3+offsets[2])` evaluated on each vertex's normalized position — to get smooth, continuous surface blotches (craters, oceans, lava cracks, flare patches) from vertex colors alone, without a texture/noise library. Per-vertex independent randomness was rejected early on since it reads as uncorrelated static rather than a texture.
- `buildPlanetMesh` picks one of five archetype builders (`buildRocky`, `buildGasGiant`, `buildIce`, `buildLush`, `buildVolcanic`) seeded off `hashString(body.id)`, so the same body always renders the same way; moons always use a separate `buildMoon` (small cratered rock only — bands/oceans/lava don't make sense at that scale). Archetypes combine the noise field with `jitterGeometry` (radial vertex displacement + `computeVertexNormals()`) for bumpiness, except the gas giant, which stays smooth and uses latitude-based sine bands instead of the noise field.
- ~3% of planets (never moons) also get a tilted ring (`buildRing`, a banded-color `RingGeometry` disc) — like the archetype choice, this is decided by drawing one more value off the same per-body `rng` after the base mesh is built, not a persisted field, since it's purely cosmetic and the rng sequence is already deterministic per `body.id`.
- `buildStarMesh`/`updateStarMesh` in `starMesh.js`: each star's core is a `MeshBasicMaterial` (self-lit, ignores scene lighting) over a noise-jittered, vertex-colored icosahedron so it reads as roiling plasma rather than a flat gem, plus two translucent additive corona shells. `buildSingleStar` multiplies every `STAR_TYPE_PARAMS` radius by `STAR_SIZE_SCALE` (2.5x) in one place rather than editing each type's range — corona scales are already relative to radius so they grow proportionally for free. `buildTurbulentSurface`'s bump amplitude and hot-color contrast (`Math.pow(n, 0.7)` instead of a linear lerp) were both raised, and the corona pulse amplitudes in `updateStarMesh` bumped up, for a more fiery/dramatic look across every star type, not just the binary ring below. A system's star is one of `STAR_TYPES` (`mainSequence`, `redDwarf`, `whiteDwarf`, `giant`, or `binary`), picked by hash off `system.id`; each non-binary type is just a different `STAR_TYPE_PARAMS` entry (radius range, hue, corona opacity/scale) feeding the same `buildSingleStar` builder. `binary` calls `buildSingleStar` twice (from `BINARY_COMPONENT_TYPES`, which excludes `giant`/`binary` to avoid runaway recursion/scale); the bigger of the two stays stationary at the system origin (the "primary") and the smaller one orbits it — real binaries orbit a shared barycenter, but a single stationary anchor is simpler and keeps every other body's already-origin-relative position untouched. Separation is sized off *corona* reach (`(r0+r1)*2.2`, not just core radius), since corona shells extend up to ~1.9x a star's radius and a tighter separation just blends the two into one blob. `updateStarMesh(mesh, elapsed, dt)` — called each frame from `main.js` with `gameState.simTime` — iterates `mesh.userData.stars`, spinning each star's own core and pulsing its own corona; only the smaller star carries an `orbit` (the bigger one has none, so it just sits still). A binary pair also gets a fiery `energyRing` (`buildEnergyRing`): a thick additive torus, vertex-painted with the same value-noise trick as the star cores (deep red through hot yellow-orange) rather than a flat tint, lying flat at the exact radius the smaller star orbits at, plus several bright "spark" spheres traveling around it on independent phases. Since the smaller star always sits at that same radius, it always intersects the ring as it goes around. The ring's fire pattern is spun each frame via `rotateOnWorldAxis` (not `rotation.y` — its local axes are already reoriented by the initial flat-lie rotation) for a flowing-plasma look; this whole ring is otherwise *static* once built (only pulses/spins in place, no per-frame repositioning), unlike an earlier straight-beam version that had to be re-measured and re-oriented between the two stars' positions every single frame.
- `main.js`'s `loadBodiesForCurrentSystem()` removes/clears any existing `starMesh`/`bodyMeshes` itself before rebuilding, rather than trusting each caller to do that first — a caller that forgot (or a future one that does) is exactly how two systems' stars/bodies could end up stacked in the same scene at once (observed as an extra-stars bug during development). Its two current callers (`clearSession`, the hyperspace jump handler) still also clear it themselves for the other per-system state (`npcMeshes`, `projectileMeshes`, `impactFlashes`) it doesn't own — that redundancy for `starMesh`/`bodyMeshes` specifically is now harmless belt-and-suspenders, not load-bearing.
- `stationMesh.js` exports both the original `buildStationMesh()` (unchanged, zero-arg, always the same ring-and-arms shape) and a new `buildStationMeshForBody(body)` that hashes `body.id` to pick one of `STATION_VARIANTS`: 4 hand-crafted archetypes (`buildStationMesh` itself, `buildSpireStation`, `buildModularStation`, `buildDomeStation`) plus 40 more generated by `buildParametricStation` — a combiner over a small core-shape pool (box/cylinder/cone/sphere/torus) and accessory pool (ring/arms/spikes/modules/stack/panels). Each of the 40 has its own fixed `structureRng` seed (so archetype *N* always has the same silhouette everywhere), while colors still vary per-instance via the body's own `colorRng` — the same hand-crafted-vs-generated split `data/shipClasses.js` uses for the ship roster. `buildStationMesh()`'s exact structure (particularly a child mesh named `'ring'`) must stay stable because `main.js`'s main-menu flyby (`updateMenuBackground`) calls `.getObjectByName('ring')` on it directly — the flyby always calls the zero-arg function, never `buildStationMeshForBody`, so it's unaffected by the new variety.
- All station archetypes (hand-crafted and generated) carry pulsing nav-beacon lights (`addBeacon`: a small emissive sphere + a bigger additive "glow" halo, same corona-shell technique as the star) and most carry a soft additive glow ring/halo (`addGlowRing`) around their most prominent circular feature. `buildStationMeshForBody` also sets a small per-body `userData.spinSpeed` on the returned group. `updateStationMesh(mesh, elapsed, dt)` — called each frame in `main.js` over every entry in `bodyMeshes` — applies that spin and pulses each beacon's glow opacity; it's a no-op for any mesh without that userData, so it's safe to call unconditionally. Since it only checks `userData.spinSpeed`/`userData.beacons` rather than anything station-specific, `buildPlanetMesh` also sets `userData.spinSpeed` on planet/moon meshes to get a free, slow axial spin out of the same per-frame loop.
- Moons visually orbit their parent planet: `main.js`'s `loadBodiesForCurrentSystem` derives each moon's orbit radius/starting angle from its already-generated position relative to its `parentId` parent (no extra persisted field), picks an orbit speed off a hash of the moon's id, and stores it in the module-level `moonOrbits` map. Every frame, `animate()` recomputes that moon's `body.position` (not just its mesh) from the orbit and writes both back — real movement of the authoritative game-state position, not a purely cosmetic mesh offset, so collision/docking/probing/radar (which all read `body.position` fresh each frame) stay consistent with what's rendered. The orbit is a flat circle at a *constant* XZ-plane radius (only the angle advances, `orbit.y` never changes), so `galaxy.js`'s `localPositionNearBody` clamps that radius at generation time to clear `parentRadius + moonRadius + MOON_ORBIT_CLEARANCE_MARGIN` — guaranteeing clearance once at generation guarantees it for the entire orbit, since the radius never shrinks afterward.
- `render/nebula.js`'s `createNebula()` scatters several large, softly-glowing additive sprite clusters (a Canvas-2D-generated radial-gradient texture, tinted per cluster) around a few random centers, so the starfield background reads as nebula/cosmic dust rather than empty black. Cluster centers are placed at a *minimum* radial distance from world origin (~2800+), not just anywhere in a bounding box — every system shares the same (0,0,0) local origin (see below), so a cluster placed too close would loom over and wash out every system's view; keeping it beyond the largest local play volume guarantees it only ever reads as distant backdrop. Not seeded off the galaxy PRNG (same convention as `starfield.js`'s plain `Math.random()` — it's fixed decoration, not gameplay-relevant content). `updateNebula(nebula, dt)` gives it a very slow constant rotation for a subtle "swirl"; called unconditionally at the very top of `main.js`'s `animate()`, before any of the early-return branches (menu/jump/dock/paused), so it keeps drifting in every game state.

### Collision

`game/collision.js`'s `resolveBodyCollisions` runs every frame against the player ship only (not NPCs): flying into a planet/moon/asteroidField/station/settlement pushes the ship back to the surface and cancels the inward velocity component (a slide, not a bounce or damage). Planet/moon/asteroidField use `body.radius`; station/settlement use fixed per-kind constants since their mesh size never varies. `main.js`'s dock-range check (`dockRangeFor`) has to stay wider than a body's collision shell (`collisionRadiusFor(body) + shipRadius`), or docking would be physically unreachable — this bit the first pass (station collision radius alone pushed the ship outside the old flat `DOCK_RANGE`).

### Docking animation

Docking/undocking is a scripted two-half position lerp (`main.js`'s `dockEffect` state machine, mirroring `jumpEffect`'s pattern): fly from wherever the ship is to just outside the body, swap the *entire* exterior scene (bodies/star/NPCs/projectiles) out for a shared generic bay interior (`render/stationInterior.js`) behind a brief flash — the flash (the same `jumpFlashEl` reused for hyperspace, recolored per effect) masks the cut — then glide to the parked spot. Undocking reverses the exact same path; `dockedApproach` remembers the original approach vector so the reverse trip lines up. The interior lives at a fixed coordinate far outside any real system's range (`DOCKING_BAY_ORIGIN`), so meshes are only added/removed from the scene, never rebuilt.

### Main menu

`ui/menu.js`'s title (`h1`) uses `drop-shadow` filters (not `box-shadow`, so the glow hugs the actual glyph shapes) animated through several hues for a slow "gaseous" halo, plus `::before`/`::after` pseudo-elements duplicating the text with opposite `clip-path` bands, `hue-rotate` filters, and `mix-blend-mode: screen` — normally invisible (`opacity: 0`), briefly flashed on by `steps(1)` keyframes for ~1 second out of every ~6.5s loop for an occasional chromatic-aberration glitch rather than constant noise.

### Controls: mouse-aim flight mode

Flight is mouse-aim, toggled by Space (`main.js`'s `flightMode`), not always-on:

- Entering flight mode calls `renderer.domElement.requestPointerLock()`; if that promise rejects, `flightMode` is set back to `false` immediately in the `.catch()` rather than waiting on the uncertain-timing `pointerlockchange` event (a `pointerlockchange` listener still syncs `flightMode` to `false` if the lock is released some other way after a successful acquisition, e.g. the browser's own forced-unlock behavior). Exiting flight mode (`exitFlightMode()`) explicitly hides the crosshair too, since `updateCrosshair()` only runs in the per-frame gameplay branch of `animate()` — it never gets a chance to hide itself if a menu opens and that branch stops running (bit us once: crosshair stayed frozen visible behind the pause menu until `exitFlightMode` was made to hide it directly).
- Opening the pause menu, nav map, or the docking UI all call `exitFlightMode()` so the OS pointer is available for clicking UI — per-instruction, entering any menu forces pointer mode. Resuming/closing a menu does *not* automatically re-enter flight mode; the player presses Space again.
- `game/flight.js`'s `updateFlight(shipState, shipClass, keys, mouseAim, dt)`: mouse deltas (`createMouseAimState()`, accumulated only while `document.pointerLockElement` is set, consumed/reset every call) directly rotate the ship's yaw/pitch, scaled by that ship class's `turnRate`; pitch is inverted (mouse up → nose up, the traditional flight-stick mapping). Q/E still roll on keys (a 2-axis mouse can't drive a third axis). W/S ramp a persisted `shipState.throttle` (-1 to 1) rather than applying raw per-frame thrust, so releasing both keys holds the current cruising speed instead of coasting to a stop; below zero, S applies reverse thrust, separately capped (via the forward-component of velocity, not the general speed clamp) to `REVERSE_SPEED_FRACTION` (25%) of the ship's forward max speed. A/D are direct lateral (strafe) thrust, independent of throttle.
- Firing is mouse-button-specific and gated by `flightMode`: left-click fires only `laser`-type hardpoints, right-click only `missile`-type ones, via `combat.js`'s `fireProjectile`'s optional `weaponTypeFilter` param (ships without a matching hardpoint simply fire nothing — no special-casing needed). NPC calls omit the filter and still fire every hardpoint, unchanged. `contextmenu` is prevented globally so right-click doesn't pop the OS context menu. Both keys and mouse deltas are only applied to `updateFlight` while `flightMode` is true (`flightMode ? keys : EMPTY_KEYS`); outside flight mode the ship still coasts under existing damping (physics doesn't pause), it just stops responding to input.
- The crosshair (`updateCrosshair()`) projects a fixed point ahead of the ship (`CROSSHAIR_DISTANCE`) the same way `updateWaypointIndicator()` projects the waypoint arrow — it shows the ship's actual instantaneous forward vector (where fixed hardpoints will fire), not the camera's, since the chase camera lags behind via `lerp` and can visibly diverge from the ship's facing during fast mouse turns.
- The HUD's velocity gauge (`hud.js`'s `.bar.velocity`) is signed and center-zero: it's fed the *forward component* of velocity (`velocity.dot(shipForward)`, computed in `main.js`), not the unsigned speed magnitude shown in the text readout next to it — fill extends right (green) for forward, left (amber, `.reversing` class) for reverse.
- Known gap: Pointer Lock's engagement itself could not be verified end-to-end via CDP-scripted testing — Chromium requires genuine OS-level window focus for `requestPointerLock()` to succeed, which an automated/background-launched Electron window may not have (`WrongDocumentError` observed in that environment specifically). The `flightMode` state, mouse-button-to-weapon-type routing, throttle/reverse/strafe/roll math, and menu-forces-pointer-mode behavior were all confirmed live (the weapon-routing check used a direct `flightMode` override to sidestep the broken pointer lock, since its rejection now resolves fast enough to flip `flightMode` back off within a single frame); the actual pointer-capture UX should be spot-checked by a human running the packaged app normally.

### Targeting

Tab locks a combat/scan target — ships (NPCs), stations/settlements, and individual asteroids within a field; planets and moons are deliberately excluded (they're dock/probe candidates, never a target). State lives in `main.js`'s module-level `currentTarget`, never persisted (NPCs aren't either): `{ kind: 'npc'|'body', id }` for ships/stations, or `{ kind: 'asteroid', fieldId, index }` for a specific rock, since one asteroid field body produces many targetable rocks rather than one.

- `render/asteroidFieldMesh.js`'s `getAsteroidRocks(body)` is the single source of truth for a field's scattered rock positions (radius/position/rotation/scale, all derived from `mulberry32(hashString(body.id))`) — `buildAsteroidFieldMesh` calls it to build the visible rocks, and `main.js`'s targeting code calls the exact same function to compute the same rocks' world positions, so a targeted reticle always lines up with the rock actually rendered.
- First Tab press (or if the current target's gone — destroyed NPC, left the system) calls `cycleTarget()`'s "acquire" path: among all in-range targetable entities, it picks whichever is most aligned with the ship's actual forward vector (`forward.dot(direction) > ~0.94`, a ~20deg forgiving cone) rather than a precise raycast — i.e. "what's under the crosshair", since the crosshair itself just projects that same forward vector (`CROSSHAIR_DISTANCE` ahead).
- Subsequent presses (target still valid) instead sort all in-range entities by distance and jump to the one after the current target, wrapping around — "cycle between nearby targets". `sameTarget`/`toTargetRef` centralize the (fieldId, index) vs. plain id comparison so this logic doesn't need to special-case asteroids inline.
- `resolveTarget()` looks the target up fresh every frame (never cached), so a destroyed NPC correctly resolves to `null` next frame rather than holding a stale position; it also returns `isAsteroid`, which both `updateTargetIndicator` (reticle tint) and `updateMiningBeam` (auto-fire eligibility — see Mining mode) key off of.
- `updateTargetIndicator()` projects the resolved target's position the same way `updateCrosshair`/`updateWaypointIndicator` do, and shows a bracket + name/distance/hull% label (hull% only for NPCs) tinted red for hostiles (`isHostileToPlayer`, see Combat AI below), amber for asteroids, or neutral blue otherwise; hidden when the target is behind the camera. Hidden explicitly in `exitFlightMode()` too, the same fix already needed for the crosshair (see above) — a per-frame update that only runs in the gameplay branch of `animate()` can't hide itself once a menu stops that branch from running.

### Combat AI: factions, pirate difficulty, and the pirate/alien truce

- NPC factions are `pirate`, `alien`, or `trader`. Traders never fight (`opponentsFor` in `combat.js` returns no opponents for them). Aliens are always hostile to both the player and pirates. Pirates are hostile to the player *unless* a truce is in effect (see below), in which case they target aliens instead.
- `combat.js`'s `opponentsFor(npc, gameState)` is the single place that decides who an NPC is currently fighting — `updateNpcAI` picks the nearest entry from it (instead of always hardcoding the player) to drive the existing patrol/attack/flee state machine, so the same state machine works for NPC-vs-player and NPC-vs-NPC without duplicating it.
- `fireProjectile`'s optional `targetRef` (`{ kind: 'player' }` or `{ kind: 'npc', id }`) records what an NPC's shot is actually aimed at, since `updateProjectiles` can no longer assume every non-player projectile is headed at the player — the player's own shots omit it and keep hitting any NPC in range, unchanged.
- `combat.js`'s `truceActive(gameState)` is simply "is there a live alien in this system" — it doesn't track discrete "arrival" events, since a pirate/alien encounter already forming, an alien arriving mid-pirate-fight, and a pirate arriving mid-alien-fight all reduce to the same continuous condition once phrased this way. `updateCombatFlag` (the no-save-during-combat gate) and `main.js`'s `isHostileToPlayer` (radar dot color, target reticle tint) both key off the same truce state so a truced pirate doesn't read as hostile to the player.
- `game/spawner.js`'s `pickPirateShipClass` sizes pirate difficulty off `coreFraction` (the same value driving the population/alien gradient — see Procedural generation): it draws from a band of the ship roster sorted by price, sliding from the cheapest/weakest third near the galactic core to the priciest/toughest third at the rim, so pirates get harder to fight the further out you go.
- `main.js`'s per-frame loop edge-detects the truce ending (`truceWasActive` was true, `truceActive` is now false — i.e. the aliens are all dead or gone) and has any surviving pirates leave: it shows a toast ("The pirates thank you...") and removes them from `gameState.npcs` directly, rather than marking them `destroyed`, so a truced pirate that happens to be an active bounty target doesn't auto-complete that mission — it just re-materializes next time the player re-enters the system (the same existing respawn path `ensureBountyNpcsForSystem` already uses).
- `updateNpcAI`'s `attack` state has two variations layered on top of the base "face and close on the opponent" behavior, both driven by per-NPC flags lazily set once (`??=`) rather than re-rolled every frame: `isOrbiter` (~50% of attackers) circles the target once within `ORBIT_RANGE` instead of always closing head-on — thrust goes mostly tangential while facing (and firing) still tracks the opponent directly, via a perpendicular vector (`(-toOpponent.z, 0, toOpponent.x)`) scaled by a per-NPC `orbitDir` (+1/-1) picked once for a consistent circling direction. Independently, any NPC hit within the last `EVADE_WINDOW_S` (checked off the existing `lastHitAt`, no new field) thrusts backward instead of closing in — a brief break-off after taking a hit, still facing (and still able to fire at) the opponent while it retreats.
- The moment an NPC's hull first drops below `FLEE_HULL_FRACTION`, it rolls once (not every frame — see the comment in `combat.js`) between the existing `flee` state and a new `ram` state (`RAM_CHANCE`, ~3%): a suicide run aimed squarely at the *player* specifically (not `opponentsFor`'s pick — a truced pirate's opponent is an alien, but a ram is always about the player), turning/accelerating faster than normal and dealing `RAM_DAMAGE` plus destroying itself on contact. `main.js`'s NPC loop shows one of `RAM_QUOTES` ("Ramming speed!", etc.) via the faction toast (recolored red for this, vs. its green default for the pirate-truce message) the first frame `aiState === 'ram'`, guarded by a one-shot `npc.ramAnnounced` flag.

### Mining mode

`R` toggles `main.js`'s module-level `miningMode`, which swaps the laser's behavior entirely rather than adding a new input: with it on, the laser ignores `laserFireHeld`/crosshair aim and instead auto-fires a continuous beam at the current target for as long as it's an asteroid, in range, and the mining hold isn't full; with it off, lasers fire exactly as before (manual left-click, toward the crosshair). Missiles are unaffected either way.

- `updateMiningBeam()` doesn't spawn physical projectiles — there's nothing for a beam to travel toward or collide with; instead it directly ticks `mineAsteroidField` on a timer (`MINING_TICK_INTERVAL_S`) whenever `miningMode && flightMode && !cruising && target.isAsteroid && inRange && !isMiningHoldFull()`, reusing the existing `onProjectileHit` toast/sound/flash feedback path used by manual mining hits.
- It stops on its own once the hold is full simply by never activating in that state (checked via `isMiningHoldFull()` *before* ticking), rather than firing a doomed tick and spamming the "hold full" toast every interval.
- The beam mesh (`buildMiningBeamMesh`) is a constant unit-length cylinder rescaled/reoriented between the ship's first laser hardpoint and the target's position every active frame — the same stretch-a-unit-shape technique the binary stars' old energy-trail beam used (see above) — so "no need to point at the asteroid" is literal: the beam bends to the target regardless of the ship's actual facing.
- A small `MINING MODE` HUD indicator (top-left, independent of the top-center supercruise indicator since the two toggles are unrelated and can both be true at once) gives a persistent visual reminder the mode is active, since the beam itself is only visible when it's actually firing.

### Save/load & permadeath

- NPCs and projectiles are never persisted — encounters are ephemeral and re-spawn procedurally on load or hyperspace jump (`missions.js`'s `ensureBountyNpcsForSystem` re-materializes any active bounty's target).
- Saving throws if `gameState.inCombat` is true (`game/save.js`).
- On hull ≤ 0, `handlePlayerDeath()` in `main.js` ends the session and shows the death screen; it does not delete the save file.

## Testing conventions

Tests use `node:test` + `node:assert/strict` only, colocated as `*.test.js` beside the module under test — no framework, no config, no mocking layer. Keep new tests in that style.
